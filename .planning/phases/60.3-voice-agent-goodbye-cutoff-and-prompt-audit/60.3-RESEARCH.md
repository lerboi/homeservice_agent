## Summary

Research is narrowly focused on the 18 questions the CONTEXT left open for the planner. The installed `livekit-agents==1.5.1` source confirms clean public hook points for every timestamp Stream A needs — no monkey-patching required. `AudioOutput` is a `rtc.EventEmitter` exposing `"playback_started"` and `"playback_finished"` events as stable public API; `AgentSession` already exposes `speech_created`, `conversation_item_added`, `close`, and a typed `CloseEvent.reason` enum that distinguishes `PARTICIPANT_DISCONNECTED` from `USER_INITIATED`/`JOB_SHUTDOWN`. The audio-frame timestamp can be captured by wrapping `context.session.output.audio.capture_frame`.

`src/prompt.py` is 634 lines with 16 section builders as CONTEXT claims, but only 3 are locale-aware via conditional Spanish branches (`_build_info_gathering_section`, `_build_booking_section`, plus the greeting/customer account sections that use `t()` message lookups). The Spanish messages JSON is parity-complete at the JSON key level — every key present in `en.json` exists in `es.json` — but the `_build_*` builders embed most Spanish prose as inline string literals inside Python conditionals, NOT as message-JSON keys. This is the real audit surface for Stream B D-B-03 (locale parity drift): prose drift can happen inside Python without touching either JSON file.

The upstream situation is now clearer than it was in 60.2. **Issue #4486** (what the 60.2-RESEARCH.md cited) is NOT the right upstream reference for the goodbye cutoff — it's a different class of segment synchronizer race (resume/pause-after-close on interruption-heavy TTS). The **authoritative upstream issue is #5096** (`"Gemini Realtime EndCallTool not working"`, filed 2026-03-12, OPEN, labeled `bug`, no linked PRs). #5096 reproduces the exact symptom: Gemini Live + EndCallTool + `"_SegmentSynchronizerImpl.playback_finished called before text/audio input is done (text_done:false, audio_done:true)"` + truncated final audio, at ~20% reproduction rate. No upstream workaround posted yet.

**Primary recommendation:** Stream A Plan 1 uses the public `AudioOutput` event emitter + a `capture_frame` wrapper + `CloseEvent` for all 6 timestamps (no private API access). Store per-call state in a single-element-list closure variable `diag_record = [{}]` mirroring the existing `call_end_reason[0]` pattern at `src/agent.py:189`. Flush inside `_on_close_async` BEFORE `run_post_call_pipeline` — verified shutdown ordering places `_on_close_async` strictly AFTER the participant-disconnect-triggered `_SegmentSynchronizerImpl.mark_playback_finished`, so the record is complete by the time we log. Stream B opens with the authoritative `src/prompt.py` section map below (CONTEXT's line numbers match), and the locale-parity audit must check Python string literals, not just JSON keys.

## Architecture Patterns

### Stream A: Goodbye Cut-off Instrumentation

The 60.2 UAT log signature — `_SegmentSynchronizerImpl.playback_finished called before text/audio input is done (text_done:false, audio_done:true)` immediately before a `CLIENT_INITIATED` participant disconnect — maps to a specific, known upstream bug (`livekit/agents#5096`, OPEN since 2026-03-12). Hooks below use only **public event emitters**, matching Voco's existing `session.on("conversation_item_added")` pattern at `src/agent.py:260`.

#### R-A1: `last_text_token_at` hook — `"conversation_item_added"` event

`[VERIFIED: events.py:90-103 + 169-172]` — `AgentSession.on("conversation_item_added")` fires for every finalized agent or user utterance carrying a `ChatMessage` with `text_content`. Voco already uses this exact event to build `transcript_turns` at `src/agent.py:260-269`. The cleanest extension: reuse the same handler; when `role == "agent"` and the chat text looks like a goodbye segment (heuristic: contains any of `["thank you for calling", "have a great", "goodbye", "bye"]` — not exact, see §R-A8 on heuristic fragility), capture `int(time.time() * 1000)` into `diag_record[0]["last_text_token_at"]`. Overwrite-on-each-turn is safe: the last agent turn before `end_call` IS the goodbye.

Alternative considered and rejected: hooking `session.output.transcription.capture_text()` at the chunk level would give a truer "last text token" but requires subclassing `TextOutput` and attaching a custom instance via `session.output.transcription = ...`, which invalidates Voco's existing room-io wiring. The `conversation_item_added` boundary is close enough for race-diagnosis purposes (we care about order of events across ~100ms intervals; the last turn's finalization is within tens of ms of the last token).

`[VERIFIED: events.py line 170]` — `ConversationItemAddedEvent.created_at: float = Field(default_factory=time.time)` — the framework also provides a timestamp on the event itself; prefer `event.created_at * 1000` for millisecond consistency with other records.

#### R-A2: `last_audio_frame_at` hook — wrap `session.output.audio.capture_frame`

`[VERIFIED: io.py:131 + io.py:241]` — `AudioOutput` is an abstract base that's also a `rtc.EventEmitter[Literal["playback_finished", "playback_started"]]`. Its `capture_frame(frame: rtc.AudioFrame)` is the call that every agent audio frame flows through before reaching the SIP participant. To capture the timestamp of the last frame emitted BEFORE playback_finished fires, the minimal non-invasive pattern is a `__wraps__` wrapper installed right after `await session.start(...)` (where `session.output.audio` is guaranteed non-None):

```python
original_capture_frame = session.output.audio.capture_frame

async def _timed_capture_frame(frame):
    diag_record[0]["last_audio_frame_at"] = int(time.time() * 1000)
    return await original_capture_frame(frame)

session.output.audio.capture_frame = _timed_capture_frame  # type: ignore[method-assign]
```

Overwrite-on-each-frame is correct — we want the most recent frame's timestamp. No subclassing, no monkey-patch of the shared module, no touching `io.py`. `[VERIFIED: pattern is idiomatic for instrumentation, mirrors the existing `@session.on(...)` model]`.

`[ASSUMED]` Risk: if `session.output.audio` is set to `None` anywhere during session cleanup (it IS — see `agent_session.py:922`), the wrapper is orphaned harmlessly. If a future refactor replaces `audio` at runtime, the wrapper breaks silently. MEDIUM risk for a diagnostic-only feature; acceptable.

#### R-A3: `playback_finished_at` + `text_done`/`audio_done` flags — `AudioOutput.on("playback_finished")` event

`[VERIFIED: io.py:131, 149-169, 182-209]` — `AudioOutput` emits `"playback_finished"` with a `PlaybackFinishedEvent(playback_position, interrupted, synchronized_transcript)` on every segment completion. This is a stable public API — `AudioOutput` inherits from `rtc.EventEmitter[Literal["playback_finished", "playback_started"]]`.

However, the specific log line the CONTEXT cites (`_SegmentSynchronizerImpl.playback_finished called before text/audio input is done (text_done:false, audio_done:true)`) is emitted from `transcription/synchronizer.py:268-279` — a private class's `mark_playback_finished` method. The warning fires ONLY when `text_done` and `audio_done` are not both true. The public `"playback_finished"` event fires UNCONDITIONALLY after the segment closes.

**Recommendation:** subscribe to the public event for the timestamp; infer `text_done=false, audio_done=true` from the PRESENCE of the warning in the log stream (read-only; the `logging.Logger` infrastructure already captures it to Railway logs). To attach the flags to the diagnostic record, add a `logging.Handler` that filters on `"_SegmentSynchronizerImpl.playback_finished called before text/audio input is done"` substring and records `text_done`/`audio_done` from `record.text_done` / `record.audio_done` (the warning passes them via `extra=`, verified at `synchronizer.py:277`).

```python
class _GoodbyeDiagHandler(logging.Handler):
    def emit(self, record):
        if "playback_finished called before text/audio" in record.getMessage():
            diag_record[0]["playback_finished_at"] = int(time.time() * 1000)
            diag_record[0]["text_done"] = getattr(record, "text_done", None)
            diag_record[0]["audio_done"] = getattr(record, "audio_done", None)

logging.getLogger("livekit.agents").addHandler(_GoodbyeDiagHandler())
```

If the warning DOESN'T fire on a given call, `playback_finished_at` is populated from the public event instead (fallback via the EventEmitter).

`[VERIFIED: synchronizer.py:273-278]` — the `extra` dict exactly contains `text_done` and `audio_done`, both bools, both picked up on `LogRecord` attributes via standard `logging` semantics.

#### R-A5: `participant_disconnect_at` + `disconnect_reason` — `session.on("close")` + `CloseEvent`

`[VERIFIED: events.py:229-244]`:

```python
class CloseReason(str, Enum):
    ERROR = "error"
    JOB_SHUTDOWN = "job_shutdown"
    PARTICIPANT_DISCONNECTED = "participant_disconnected"
    USER_INITIATED = "user_initiated"
    TASK_COMPLETED = "task_completed"

class CloseEvent(BaseModel):
    type: Literal["close"] = "close"
    error: (...) | None = None
    reason: CloseReason
    created_at: float = Field(default_factory=time.time)
```

`AgentSession.on("close", handler)` → `handler(event: CloseEvent)`. The `event.reason` gives one of the 5 enum values above. For Voco, caller hangup → `PARTICIPANT_DISCONNECTED`. A `rtc.DisconnectReason` (CLIENT_INITIATED vs SERVER_INITIATED) distinction is NOT exposed through `CloseEvent` directly — it's logged by `room_io.py:414-421` via `rtc.DisconnectReason.Name(participant.disconnect_reason)` but NOT passed to the close event. The 60.2 UAT log `"closing agent session due to participant disconnect reason=CLIENT_INITIATED"` comes from that room_io logger line.

**To capture `disconnect_reason` specifically** (which CONTEXT D-A-02 requires), hook `ctx.room.on("participant_disconnected")` at the room level. This is the same event `room_io.py:175` subscribes to:

```python

        diag_record[0]["participant_disconnect_at"] = int(time.time() * 1000)
        diag_record[0]["disconnect_reason"] = rtc.DisconnectReason.Name(
            participant.disconnect_reason or rtc.DisconnectReason.UNKNOWN_REASON
```

#### R-A6: Per-call mutable diagnostic record — single-element-list closure

CONTEXT offered three options:
1. `deps`-injected dict (existing `deps["_tool_call_log"]`, `deps["_booking_succeeded"]` pattern at `src/agent.py:211`)
2. Single-element-list closure (existing `call_end_reason = ["caller_hangup"]` pattern at `src/agent.py:189`)
3. Module-level `WeakKeyDictionary[AgentSession, dict]`

**Recommendation: use BOTH #1 and #2 together, deliberately.** Seed `deps["_diag_record"] = [{"started_at": int(time.time()*1000)}]` at line 212 of `agent.py` (right next to `"_tool_call_log": []`). The tool `end_call.py` writes into `deps["_diag_record"][0]` — that's #1. The session-level handlers in `agent.py` close over the same `deps["_diag_record"]` via the entrypoint scope — that's #2 semantically (a mutable reference captured in a closure) but physically #1 (lives on deps).

**Reason:**
- Matches 60.2-RESEARCH.md Pitfall 2: module globals break under concurrent calls on the same worker. Both `deps` and closure-captured lists are per-session.
- Matches the established pattern for `call_end_reason` which is BOTH on `deps["call_end_reason"]` AND referenced by the entrypoint closure as `call_end_reason`. Tools read/write it via `deps`, session-level code reads/writes it via the closure reference — the same list. `[VERIFIED: src/agent.py:188-189 + src/tools/end_call.py:60]`.
- Option #3 (WeakKeyDictionary) adds a module-level dependency for zero benefit on this phase's scope.

#### R-A7: Diagnostic flush site — `ctx.add_shutdown_callback(_on_close_async)` at agent.py:363

**Shutdown ordering (verified):**

1. Caller hangs up → LiveKit room emits `"participant_disconnected"` → `room_io.py:403-432` handler runs → emits the `"closing agent session due to participant disconnect"` log line.
2. Room-io invokes `self._agent_session.aclose()` (`room_io.py:425` approx — see the `_close_session_atask` setup).
3. `AgentSession.aclose()` drains speeches (`agent_session.py:906-913`) → `_SegmentSynchronizerImpl.mark_playback_finished` fires on each remaining segment (this is where the warning originates) → `synchronizer.py:268-279`.
4. Session fully closed → `session.emit("close", CloseEvent(reason=PARTICIPANT_DISCONNECTED, ...))`.
5. JobContext sees the session closed → runs shutdown callbacks registered via `ctx.add_shutdown_callback(...)` → `_on_close_async` runs. `[VERIFIED: agent.py:363 registration + SDK docs for JobContext.add_shutdown_callback]`.

**Conclusion:** by the time `_on_close_async` starts, `diag_record[0]` already contains `playback_finished_at`, `text_done`, `audio_done`, `participant_disconnect_at`, `disconnect_reason`, and `session_close_at`. The flush site at the TOP of `_on_close_async` (before the 8s `asyncio.wait_for(run_post_call_pipeline, ...)`) captures a complete record. Emit `logger.info("[goodbye_race] %s", json.dumps(...))` and `sentry_sdk.add_breadcrumb(...)` right there. `[VERIFIED: ordering via installed source]`

One subtlety: `_on_close_async` runs inside the 8s timeout wrapper. The flush must be the FIRST thing the function does so a timeout mid-pipeline doesn't lose the diagnostic log. If `run_post_call_pipeline` times out, the flush already happened; the TIMEOUT warning appears SEPARATELY in the log (as in the 60.2 UAT evidence), so forensic search on `[goodbye_race]` still finds the record.

#### R-A8: `[goodbye_race]` log entry schema (recommended)

Single JSON object per call. Keys:

```json
{
  "schema_version": 1,
  "call_id": "call-_+6587528516_ABC123",
  "tenant_id": "uuid-or-null",
  "caller_phone_sha256": "first-16-hex-chars",
  "started_at_ms": 1713...,
  "end_call_invoked_at": 1713... or null,
  "last_text_token_at": 1713... or null,
  "last_audio_frame_at": 1713... or null,
  "playback_finished_at": 1713... or null,
  "text_done": true|false|null,
  "audio_done": true|false|null,
  "participant_disconnect_at": 1713... or null,
  "disconnect_reason": "CLIENT_INITIATED|SERVER_INITIATED|UNKNOWN_REASON|null",
  "session_close_at": 1713... or null,
  "close_reason": "participant_disconnected|user_initiated|...|null",
  "transcript_tail": "agent: ... | user: ... | agent: ..." (last 3 turns joined with " | ", truncated to 500 chars),
  "tool_call_log_tail": ["check_availability", "book_appointment", "end_call"] (last 5 tool names from deps._tool_call_log)
}
```

**Rationale for each key:**
- `schema_version`: **REQUIRED** per D-A-03. If 60.3 iterates on schema mid-phase (likely — this is a diagnostic-first feature), grep-based parsers can version-branch.
- `caller_phone_sha256`: privacy — never log raw E.164 numbers. 16-char prefix is ample for deduplication; full hash is collision-safe but adds noise.
- `*_at` keys as `int` milliseconds: uniform across Python's `time.time()` (float seconds) and JS `Date.now()` (int milliseconds); simpler to diff than floats. All `null`-able — the instrumentation is lossy by design on some code paths.
- `transcript_tail`: 500-char cap prevents Railway log bloat (one call × 500 chars × Sentry breadcrumb + log = ~1.2 KB, well under the 50-100 KB budget CONTEXT D-A-03 sets).
- `tool_call_log_tail`: CONTEXT doesn't require this but `deps["_tool_call_log"]` already exists (see `agent.py:211`) and knowing the tool sequence immediately before goodbye clarifies whether `end_call` was really the turn that truncated.

Sentry breadcrumb uses the same dict; Sentry auto-serializes. Breadcrumb `category="goodbye_race"`, `level="info"` (or `"warning"` if `text_done=false` OR `playback_finished_at - last_audio_frame_at < 100` to elevate anomalies automatically).

**System instruction integration point:** `[VERIFIED: src/agent.py:246]`

The plugin forwards `instructions` into Gemini Live's WebSocket session config as the `system_instruction` (Vertex/Gemini parlance). Update semantics: once the session starts, `instructions` is static for the session — `session.update_instructions(...)` triggers a warning and is a no-op for `gemini-3.1-flash-live-preview` (see 60.2-RESEARCH.md Fix J verdict + commit `b2a9adf` `(gemini realtime): add warnings in update_chat_ctx and update_instructions`). **Documented length/token limits for Gemini Live:** `[ASSUMED]` Gemini 3.1 Flash Live accepts system_instruction up to the model's context window (nominally 1M tokens for 3.1 Flash; practical max is far lower because shorter system instructions → faster first-token → lower per-turn latency). The current Voco prompt is ~3500-4500 tokens (ballpark, from 634-line prompt.py rendered output). **No hard limit hit; token economy is about first-token latency, not context truncation.**

**Community-reported realtime-model prompt anti-patterns** `[ASSUMED — based on LiveKit community Discord reports + OpenAI Realtime API docs — not formally verified for Gemini Live specifically]`:

1. **Chat-LLM preamble patterns** ("You are a helpful AI assistant..." / "As an AI, I should...") cause the realtime model to speak them aloud. Voco's prompt mostly avoids this; `_build_identity_section` opens with "You are the AI phone receptionist for..." which is fine per community convention.
2. **Second-person self-address** ("you should think before responding") may cause some realtime models to narrate reasoning. Voco's prompt uses second-person throughout — need to audit for self-narration triggers.
3. **Markdown headings vs ALL CAPS labels**: Gemini Live appears to honor ALL-CAPS section labels (`VOICE & CONVERSATION STYLE:`) better than `##` markdown headers. Voco already uses ALL-CAPS — preserve this.
4. **Negated instructions** ("DO NOT say X") sometimes produce the forbidden output inverted. Prefer positive framing ("Say Y"). Voco's prompt has heavy `NOT`/`NEVER`/`DO NOT` usage in critical rules — the audit should assess whether each negation could be reframed as positive without losing the anti-hallucination strength.
5. **Length** — longer prompts increase first-audio latency on realtime models more than on chat LLMs because the full prompt is encoded on session start. Token economy matters (CONTEXT D-B-02 dim 6).

**CRITICAL RULE framing** — already present in Voco for `_build_corrections_section` + `_build_outcome_words_section` + `_build_customer_account_section`. The audit should verify these are at the **TOP** of the assembled prompt; currently they're in positions 3, 4, 10 respectively per R-B1 orchestrator order. ## Common Pitfalls

### Pitfall 1: Confusing upstream issue #4486 with #5096

**What goes wrong:** Plan cites #4486 as the upstream goodbye-cutoff reference; a future reader tries to correlate the fix with #4486's `pause/resume after close` symptoms and gets confused.
**Why it happens:** 60.2 SKILL.md + CONTEXT both cite #4486 as "the `_SegmentSynchronizerImpl` race." But #4486 is a different class (interruption-heavy TTS pause/resume) and #5096 is the Gemini-Live + EndCallTool specific one matching the UAT signature exactly.
**How to avoid:** Stream A plans cite **#5096** as the primary upstream reference; #4486 stays in the Fix J / plugin-upgrade context only.
**Warning signs:** None at runtime — this is a documentation correctness issue only.

### Pitfall 2: Wrapping `session.output.audio.capture_frame` before `session.start()`

**What goes wrong:** `session.output.audio` is `None` until room-io attaches it during `session.start(...)`. Wrapping too early fails with AttributeError.
**Why it happens:** `io.py:404-408` + `agent_session.py:649-703` — the output is attached during `session.start()` room-io handshake.
**How to avoid:** install the wrapper **after** `await session.start(...)` completes (around `src/agent.py:476`, right before the initial `generate_reply`).
**Warning signs:** `AttributeError: 'NoneType' object has no attribute 'capture_frame'` at session start → wrapper installed too early.

### Pitfall 3: `logging.Handler` chained to wrong logger name

**What goes wrong:** Diagnostic handler attached to `logger = logging.getLogger("voco-agent")` instead of `logging.getLogger("livekit.agents")` → never sees the `_SegmentSynchronizerImpl.playback_finished` warning.
**Why it happens:** the warning originates in the livekit-agents package, not Voco's. Python's `logging` propagates to root, but project convention may add filters that silently drop framework-level records.
**How to avoid:** attach to `logging.getLogger("livekit.agents")` explicitly (or to the root logger if a stricter filter-chain audit confirms no drops). Unit-test coverage: emit a synthetic `LogRecord` with the substring and assert the handler captured it.
**Warning signs:** Plan 2 UAT yields `diag_record[0]["playback_finished_at"]` null on calls where Railway log clearly shows the warning fired.

### Pitfall 4: Heuristic for "is this utterance the goodbye" is fragile

**What goes wrong:** The `"thank you for calling" OR "goodbye"` heuristic in R-A1 misses non-English-goodbye utterances OR fires on a caller's "thank you" mid-call.
**Why it happens:** text-matching against speech is inherently fuzzy; Gemini rephrases goodbyes; Spanish callers get Spanish goodbyes not in the heuristic list.
**How to avoid:** store `last_text_token_at` on EVERY agent turn (overwriting) instead of filtering by goodbye-content. The "last" by definition is the most recent one, which for `end_call`-terminated calls IS the goodbye. The heuristic is only needed if Voco wants to distinguish "goodbye turn" from "earlier turn" post-hoc — and the analysis step (Plan 2 deliverable) can do that from the `transcript_tail` without needing the hook to pre-classify.
**Warning signs:** a caller SAID "thanks" mid-call and `last_text_token_at` then reflects that point, not the final goodbye. This is actually desirable (we want the LAST token), but surprising if the engineer expected heuristic pre-filtering.

### Pitfall 5: Cross-repo Windows path (carried from 60.2)

**What goes wrong:** Plan cites `/Users/leroyngzz/Projects/livekit_agent/` (macOS underscore path from 60.2-CONTEXT line 288) or `livekit_agent/src/...` (underscore) when the actual path is `C:\Users\leheh\.Projects\livekit-agent\` (Windows hyphen).
**Why it happens:** 60.2-CONTEXT.md had the wrong path literally; 60.2-RESEARCH.md corrected it (Pitfall 5); some downstream artifacts still reference the wrong form.
**How to avoid:** all 60.3 plans and the audit doc use `livekit-agent/src/...` (hyphen) for paths; full Windows path when quoting absolute. CONTEXT.md D-A-06 already locks this.
**Warning signs:** file not found errors during execution; `git status` in wrong repo.

### Pitfall 6: Stream B patches breaking 60.2 inverted-assertion tests

**What goes wrong:** Stream B Plan 2 (tool_narration audit) modifies `_build_tool_narration_section` and inadvertently re-adds `"runtime automatically plays"` or similar — causing `test_tool_narration_does_not_claim_runtime_filler` to fail.
**Why it happens:** 60.2 Plan 05 added these specifically to guard against re-landing the reverted Fix H pattern. An audit that normalizes prose might re-introduce the banned phrases semantically.
**How to avoid:** Stream B Plan 2's diff MUST preserve the invariant that the prompt does NOT claim a runtime filler AND does instruct the model to speak its own filler. If the plan genuinely wants to change this behavior (e.g., if Gemini upgraded to support `session.say()` without TTS), it must also update the tests — and that's a red flag requiring discuss-phase escalation, not a silent change.
**Warning signs:** `pytest tests/test_prompt.py` failing after a Stream B patch.

### Pitfall 7: Sentry breadcrumb data-dict size bloat

**What goes wrong:** `transcript_tail` exceeds 500 chars, plus 10 other keys, sent every call × every breadcrumb → Sentry ingestion cost spikes; breadcrumb dropped by Sentry's 16KB limit per scope.
**Why it happens:** Sentry caps breadcrumbs at 100 per scope and each is size-limited. Log-surface is cheaper than breadcrumb-surface.
**How to avoid:** truncate `transcript_tail` to 500 chars explicitly. Emit breadcrumb only at end-of-call (one per call), not per-hook. Sentry's own docs list 8KB as a soft limit per breadcrumb.
**Warning signs:** Sentry shows "breadcrumb truncated" on goodbye_race events.
