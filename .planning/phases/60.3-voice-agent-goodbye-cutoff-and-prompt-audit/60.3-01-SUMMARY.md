## Decisions Made

- **Module-level flush helper over inline flush block:** Made `_flush_goodbye_diag` a top-level async function so pytest can invoke it directly with fixtures. The alternative (inline block inside `_on_close_async`'s closure) would require a fake `AgentSession` + `JobContext` to test.
- **Phone-redaction regex, not passthrough:** The plan spec said *"raw `from_number` string is NOT a substring of the serialized payload"*. Simplest interpretation (passthrough — just don't write from_number) doesn't cover caller-spoken numbers in transcript turns. Regex redaction is a stronger invariant; aligns with threat model T-60.3-01 beyond the explicit `caller_phone_sha256` field.
- **Filtered breadcrumb assertion in test 5:** sentry-sdk's logging integration auto-emits a breadcrumb for every `logger.info` call inside a transaction — so `mock_breadcrumb.assert_called_once()` fails with 2 calls. Filtering by `category="goodbye_race"` keeps the test pinned to our explicit breadcrumb without coupling to sentry-sdk internals.
- **Pre-existing VIP-routing test failure documented in deferred-items.md:** Confirmed via `git stash` + pre-Stream-A HEAD checkout reproduction. Pre-dates 60.3-01; touches `src/webhook/app.py` (out of Stream A scope).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Added E.164 phone-redaction to transcript_tail**
- **Found during:** Task 2 verification (test_transcript_tail_truncated_to_500_chars_and_no_raw_phone failed first run)
- **Issue:** Plan's must_haves required *"caller_phone_sha256 is used; raw E.164 from_number NEVER appears in the log payload"* and test 7 planted a raw phone inside a transcript turn that would land in the last-3-turn window. Initial implementation (pass-through tail) leaked `+6587528516` into the serialized payload.
- **Fix:** Added `_PHONE_REDACT_RE = re.compile(r"\\+?\\d{7,15}")` at module level; applied via `_PHONE_REDACT_RE.sub("[PHONE]", transcript_tail)` before writing `rec["transcript_tail"]`. Satisfies T-60.3-01 uniformly for SIP-attrs-known phones AND caller-spoken numbers.
- **Files modified:** `src/agent.py` (regex + redaction call)
- **Verification:** `pytest tests/test_goodbye_diag.py` — all 7 green; the `+6587528516` string in the test transcript turn becomes `[PHONE]` in the serialized JSON payload.
- **Committed in:** `c4f0570` (Task 2 commit)

**2. [Rule 1 - Bug] Adjusted test 5 breadcrumb assertion from assert_called_once to category filter**
- **Found during:** Task 2 verification (test_flush_is_first_in_on_close_even_if_pipeline_times_out failed first run with AssertionError: Expected 'add_breadcrumb' to have been called once. Called 2 times.)
- **Issue:** sentry-sdk's `LoggingIntegration` auto-emits an info-level breadcrumb for every `logger.info` call inside a transaction — so when `_flush_goodbye_diag` calls `logger.info("[goodbye_race] ...")` the mock sees BOTH the auto-emitted log-breadcrumb AND our explicit `sentry_sdk.add_breadcrumb(category="goodbye_race", ...)` call.
- **Fix:** Replaced `mock_breadcrumb.assert_called_once()` with a filtered list-comprehension that selects only calls where `kwargs.get("category") == "goodbye_race"` and asserts exactly one such call. Pins the test to OUR breadcrumb without coupling to sentry-sdk internals.
- **Files modified:** `tests/test_goodbye_diag.py`
- **Verification:** Test now passes while still enforcing the R-A7 invariant (goodbye_race breadcrumb fired exactly once even when pipeline times out).
- **Committed in:** `c4f0570` (Task 2 commit, alongside the main impl)
