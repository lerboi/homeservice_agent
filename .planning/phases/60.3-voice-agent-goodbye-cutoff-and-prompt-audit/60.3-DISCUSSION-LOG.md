## Area 1 — Instrumentation depth + sampling

### Q1.1 — Where do timestamps + structured fields land?

| Option | Description | Selected |
|--------|-------------|----------|
| (a) | Local stdlib `logger.info(...)` only — structured JSON dict | |
| (b) | Local logger AND Sentry breadcrumbs (`sentry_sdk.add_breadcrumb`) | ✓ |
| (c) | Local logger AND a dedicated Supabase `voice_diagnostic_events` row per call | |

**Selected:** (b) — local logger + Sentry breadcrumbs.
**Rationale:** Owner already monitors Sentry; Railway log search is for forensic deep-dives. Both surfaces serve different operational roles. DB writes on the hot path (option c) are out of proportion to a diagnostic-only feature.

### Q1.2 — Which timestamps to capture?

| Option | Description | Selected |
|--------|-------------|----------|
| (a) | Minimal: `end_call_invoked_at`, `last_text_token_at`, `participant_disconnect_at`, `session_close_at` | |
| (b) | Minimal + audio-frame markers: above PLUS `last_audio_frame_at` and `_SegmentSynchronizerImpl.playback_finished_at` | ✓ |
| (c) | Full: minimal + audio-frame + every text-delta timestamp + every VAD state change | |

**Selected:** (b) — minimal + audio-frame markers.
**Rationale:** Caller pain is *audio* truncation; without `last_audio_frame_at` and `playback_finished_at` we can't distinguish "Gemini stopped emitting text" from "audio pipeline ate the last frame" — fewer UAT iterations to a real fix means the caller-facing bug ships sooner.

## Area 3 — Prompt audit dimensions + deliverable shape

### Deliverable shape

| Option | Description | Selected |
|--------|-------------|----------|
| (a) | Findings doc + per-section patch series (one plan per section) | ✓ |
| (b) | Direct edits with inline rationale | |
| (c) | Diff with verdict table (single deliverable, no per-section atomicity) | |

**Selected:** (a) — `60.3-PROMPT-AUDIT.md` + per-section patch plans.
**Rationale:** A "big-bang" prompt rewrite makes UAT regressions impossible to triage and forces an all-or-nothing revert. Per-section atomicity keeps the owner from being whipsawed by a single bad change in a sea of good ones.

### Audit dimensions (ranked by caller-UX impact)

All 7 dimensions selected; ordering reflects priority during audit:

1. Anti-hallucination strength
2. Realtime-model-specific phrasing
3. Section ordering within prompt
4. STATE+DIRECTIVE consistency
5. VAD-redundant guidance
6. Token economy
7. Locale parity drift

**Rationale:** Caller hearing wrong info is the worst possible outcome (booking time, address, price); realtime-model misalignment causes the model to "think out loud"; section ordering matters because Gemini attends more to top of prompt; STATE+DIRECTIVE keeps the AI decisive; VAD-redundant prose competes with Fix G config; shorter prompt = faster first-token; locale parity prevents tiered UX between en/es callers.
