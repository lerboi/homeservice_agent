## Lessons captured

1. **Evidence-driven fix selection saves iteration cycles.** Plan 02's UAT #1 evidence was genuinely ambiguous (matched both #5096 signature AND Branch P directional signal). Rather than shipping both fixes or coin-flipping, the Plan-2 decision rule "ambiguous → prefer lower-risk (prompt-only, fully revertable) candidate" let Plan 03 ship Branch P with a clean discrimination signal: if UAT #2 shows the race persists unchanged, escalate to Branch G. This saved re-touching the goodbye surface twice.
2. **Public event emitters are sufficient — no monkey-patching needed.** Stream A instrumentation captured every signal it needed (including the internal `text_done`/`audio_done` flags on the `_SegmentSynchronizerImpl.playback_finished` warning) via a stdlib `logging.Handler` reading `LogRecord extra=` fields. Zero private-symbol access; zero framework patching. This pattern generalizes to any future LiveKit-agents internal-signal capture.
3. **Inline `if locale == "es":` branches scale; JSON-extraction refactor is not worth it.** The `_build_info_gathering_section` gold-standard pattern proved it: one source of truth for both locales, adjacent in code, grep-able parity count (`grep 'if locale == "es":' src/prompt.py`), and no cross-boundary templating layer. JSON-extraction would have required restructuring 15 builders + a templating accessor — out of proportion to the benefit.
4. **60.2 inverted assertions are load-bearing and outlive their triggering phase.** Plan 06 preserved the 60.2 Pitfall-6 "no runtime filler" guard-rail tests verbatim when adding the tool_narration ES branch. Without that preservation, the 60.2 Fix H revert narrative would be silently unenforced — the test's inverted assertion is the only enforcement surface. Codified as a skill rule: inverted assertions that document "this was reverted for a specific reason" are permanent invariants, not legacy.
5. **`caller_phone_sha256` adds almost no code cost and is strictly better than log-scrubbing later.** Stream A chose per-call SHA-256 (first-16-hex) of the caller E.164 number for operational grouping. Raw numbers never hit logs or Sentry. This is cheaper, simpler, and more auditable than a log-scrubbing middleware layer — apply to future diagnostic instrumentation by default.
6. **Prompt *content* (what to say) and prompt *mechanics* (how to sequence) are separate failure modes.** UAT #2's decline-case gap — agent said only "I understand." with no farewell — was mechanically correct under Branch P's CRITICAL RULE (two-step sequencing respected) but content-incomplete (no farewell phrase before end_call). The CRITICAL RULE prescribed mechanics but not content, and the model took the literal read. Audit dimension D1 was expanded to cover this class (mandate farewell content in every end_call path) in Plan 05, and the cumulative UAT confirmed the content fix held (full goodbye completed) — but the mechanical fix over-corrected (16s wait before invoking end_call). The meta-lesson: **mechanics and content are BOTH in scope, and tightening one can loosen the other**.

## Follow-ups (non-blocking)

Prioritized list. The first item is the anchor candidate for **Phase 60.4 Voice Agent Call-Teardown + Tool-Stall Filler + Availability-Intent**.

1. **HIGH — NEW 60.3 regression: 16s end_call stall after goodbye.** Branch P's CRITICAL RULE text — `"THEN in a separate turn with no additional speech, call end_call."` — over-corrected. The model now waits indefinitely for the "separate turn with no additional speech" trigger, never self-promoting to tool invocation. Evidence: cumulative UAT Call B-1 `last_text_token_at → end_call_invoked_at` gap of **16.177 seconds** on a call with a fully-completed farewell; caller said "Hello." into the dead air and manually hung up. Fix candidates: (a) rework the CRITICAL RULE to add an explicit "after the farewell, call end_call within 1-2 seconds" directive; (b) tool-side auto-teardown timer that invokes end_call if the agent has been idle for N seconds post-farewell; (c) Branch G-adjacent — after the agent's last text token fires, schedule end_call with a `wait_for_playout`-gated delay. Anchor this in Phase 60.4.

2. **MEDIUM — Tool-execution stalls (70-87s silence during `book_appointment`).** Owner-reported. Evidence: on cumulative UAT Call B-1, `check_availability` succeeded at t=13:44:26, `book_appointment` succeeded at t=13:45:53 — 87s gap with caller silence and intermittent tool_narration filler. Gemini Realtime stalls during long-running tool calls without emitting filler audio; the `_build_tool_narration_section` prompt instructs "speak filler while tools run" but model compliance is inconsistent under Gemini Realtime sync constraints. Fix candidates: (a) prompt refinement to strengthen the contract; (b) framework-level fallback filler emission if the model is silent > N seconds mid-tool-call. Both are non-trivial and warrant dedicated planning. Fold into Phase 60.4.

3. **MEDIUM — "Earliest availability" intent not handled.** Owner-reported. Caller asked `"Now, when is the earliest when is the earliest day that you have availability for?"` and the agent could not answer directly; it narrowed only after the caller supplied a specific date. The `check_availability` tool and `_build_booking_section` assume a date-narrowing flow, not an open-ended "earliest slot" search. Fix candidate: either (a) add an `earliest=true` mode to `check_availability` that returns the nearest N open slots ordered by start time, or (b) tighten the booking_section prompt to handle "earliest" queries by probing timeframes. Scope into Phase 60.4 as Stream C.

4. **LOW — Cold-start silence (10+ s at call start).** Owner-reported. LiveKit session start takes ~3.5s; Gemini Realtime cold-connection adds another ~6-7s. Infrastructure scope (not prompt-addressable). Track separately from Phase 60.4 unless it correlates with the end_call stall.

5. **LOW — STT-ASR mishearing English audio as Spanish.** Owner-reported transcript artifacts (`¿Sabes?`, `¿Qué vas a aprender?`) despite caller speaking English throughout. Gemini Realtime SDK-level issue; the agent correctly continued in English, so the language-detection branch (`_build_language_section` default-to-locale) is not the cause. Out of scope for prompt work; revisit if Gemini Realtime exposes a language-allowlist or language-hint parameter.

6. **LOW — Spanish UAT not placed (Call B-2).** Plans 05-12 ES code has shipped to Railway but was never exercised against live Gemini Realtime audio in production. Structural parity is proven at the test layer (205 passed including parity guard-rails); behavioral parity remains unverified. **Add to Phase 60.4 scope** — execute a Spanish-persona UAT call after the 60.4 teardown fix lands to simultaneously verify ES branches and the teardown fix on a locale-switched call.

7. **NON-BLOCKING — Instrumentation sample-rate dial-back (D-A-03 trigger).** Currently every-call. Originally planned to dial back to race-detected-only after N≥10 post-fix clean UAT calls. **Do NOT dial back yet** — the new 16s end_call-stall regression (item 1 above) makes full-sample `[goodbye_race]` coverage operationally valuable during the 60.4 investigation. Re-evaluate after Phase 60.4 ships and shows N≥10 clean calls with BOTH no truncation AND end_call within 2s of last text token.

8. **NON-BLOCKING — Fix I (post-call 8s pipeline timeout) monitoring.** No firing observed during 60.3 UAT. Continue passive monitoring. Promote to its own phase only if it fires on a call that is otherwise clean.

9. **NON-BLOCKING — Fix J (livekit-plugins-google upgrade past `43d3734`) re-evaluation.** No triggers fired during 60.3. Revisit only if: (a) livekit/agents#5096 closes with a fix, OR (b) Railway preview environments become available, OR (c) the team accepts switching away from `gemini-3.1-flash-live-preview`.

10. **NON-BLOCKING — Token-economy compression** opportunities documented in `60.3-PROMPT-AUDIT.md` §Cross-Cutting Concerns. Deferred.

11. **NON-BLOCKING — TONE_LABELS + customer_context STATE block locale extension.** Not caller-facing prose; tenant-config/Jobber-Xero cross-runtime compatibility. Future phase if Hispanic market data shows value.

12. **NON-BLOCKING — ES register normalization (USTED vs TÚ).** Plans 05/06/07/10/11/12 use USTED; Plans 08/09 use TÚ (flagged during Plan-08/09 execution). Not a caller-experience bug — both registers are grammatically correct Spanish. Pure code-consistency polish; queue for a dedicated future plan rather than retouching audit-passed ES prose.
