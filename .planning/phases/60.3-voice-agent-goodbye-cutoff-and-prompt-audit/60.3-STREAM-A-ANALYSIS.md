# Stream A — Goodbye Race Fix Selection

**UAT evidence source:** [60.3-HUMAN-UAT.md §"Stream A — UAT #1"](./60.3-HUMAN-UAT.md)

## Evidence summary

UAT #1 (call-_+6587528516_KwsBVWBZkKps, duration 174s, declined booking) reproduced the goodbye cutoff clearly: the caller heard the agent emit the string `"Alright, I'll get all"` and then silence before line-drop — a mid-sentence truncation of what was presumably `"Alright, I'll get all that saved down for you..."`. The `[goodbye_race]` diagnostic payload shows **`text_done=false, audio_done=true`** at the synchronizer's `playback_finished` call, which is the **textbook upstream #5096 signature** (R-A9). However, the timestamps ALSO show **`end_call_invoked_at` fired 11ms BEFORE `last_text_token_at`** (1776849313217 vs 1776849313228) — directionally consistent with "model invoked end_call before its final text token was emitted," which is the Branch P indicator. The 11ms delta is within measurement noise (event-loop dispatch latency on two independent coroutines), so the directional reading is weak on its own. Additionally, `playback_finished_at` (156.540s after call start) fired **15.2 SECONDS before `end_call_invoked_at`** (171.779s) — meaning the synchronizer's `playback_finished` event recorded on the diagnostic record was captured on an EARLIER speech segment, not the final goodbye turn. Three `_SegmentSynchronizerImpl.playback_finished called before text/audio input is done` warnings fired at +152s, +202s, and +222s during the call (not just at goodbye), indicating the pipeline-sync race is systemic across the whole call, not isolated to the farewell turn.

**Characterization:**
- end_call_invoked_at vs last_text_token_at delta: **-11ms** (end_call 11ms earlier; within event-loop noise but directionally suggests Branch P)
- end_call_invoked_at vs playback_finished_at delta: **+15239ms** (playback_finished_at captured on an EARLIER segment — stale; the final-segment playback event was never recorded before session close)
- `_SegmentSynchronizerImpl` warning fired: **yes, 3× during call** (+152s, +202s, +222s — systemic, not goodbye-specific)
- text_done at warning: **false**; audio_done: **true** (#5096 signature)
- Caller-perceived truncation severity: **last ~4-6 words cut** — heard `"Alright, I'll get all"`, lost `"...that saved down for you"` (mid-sentence cut after 5 words into the goodbye turn)

## Fix candidate options

**Branch P — Prompt-harden `_build_call_duration_section`:**
- Applicable if the model invoked end_call BEFORE finishing the farewell speech (end_call_invoked_at < last_text_token_at OR the transcript_tail shows the model's last turn was an aborted goodbye).
- Fix shape: promote the existing goodbye-protection guidance at `src/prompt.py:544-556` to a CRITICAL RULE block with an explicit STATE+DIRECTIVE failure-mode example. Move the section near the top of the assembled prompt (RESEARCH §R-B5: Gemini attends more strongly to top-of-prompt).
- Evidence required: model behavior issue visible in timestamps OR transcript.

**Branch G — Tool-level pre-guard in `end_call.py`:**
- Applicable if the model finished speaking (last_text_token_at reached end of farewell per transcript_tail) but the audio pipeline truncated the last segment (playback_finished_at fired early with text_done=false).
- Fix shape: in `create_end_call_tool`, `await session.output.audio.wait_for_playout()` before scheduling `_delayed_disconnect`. Leave the 12s sleep unchanged (scope boundary — CONTEXT). `wait_for_playout()` is a public API returning `PlaybackFinishedEvent` (RESEARCH R-A2 reference + R-A9 candidate).
- Evidence required: audio-pipeline drain race visible in the `_SegmentSynchronizerImpl` warning + playback_finished_at before last_audio_frame_at OR `text_done=false` flag.

## Selected fix

**Selected fix: Branch P**

**Rationale:** The UAT #1 evidence is **genuinely ambiguous** — it matches BOTH signatures simultaneously. The #5096 signature (`text_done=false, audio_done=true`) is present in its textbook form and 3× mid-call SegmentSynchronizer warnings prove the pipeline race is real and systemic, both of which independently argue for Branch G. But the 11ms `end_call_invoked_at` → `last_text_token_at` delta AND the fact that `playback_finished_at` in the record is stale (from an earlier segment, not the final one) means we cannot definitively prove the model emitted its full farewell text before invoking end_call — the transcript shows `"Alright, I'll get all"` as the last agent turn, which reads as an aborted/truncated generation at the model layer, not necessarily an audio-pipeline truncation of a complete text. Per the Plan 2 decision rule (lines 285-294 of `60.3-02-PLAN.md`): **ambiguous evidence → prefer Branch P**, because Branch P is lower-risk (prompt-only change, fully revertable in one commit, no runtime-behavior change to end_call's disconnect sequencing), while Branch G introduces a new `await` on a pipeline step that could in edge cases extend call teardown — a bigger behavior change that deserves clearer discriminating evidence before shipping. We commit to Branch P for Plan 3; if UAT #2 post-Branch-P shows the truncation persists unchanged (model still emits full text but audio still cuts), that cleanly discriminates to Branch G for a follow-up iteration within this phase's evidence-driven loop.

## Plan 3 execution directive

Plan 3 contains TWO bracketed branches (P and G). Execute ONLY the branch matching the selection above — **Branch P** (`_build_call_duration_section` prompt-hardening). Mark the Branch G tasks as `skipped` in the commit message. Do not execute both.

## Open questions / non-blocking concerns

- **Systemic mid-call SegmentSynchronizer warnings (3×):** The warnings at +152s, +202s, +222s are not confined to the goodbye turn and correlate with 3× `server cancelled tool calls` log lines earlier in the call. This suggests the #5096 pipeline race fires whenever a tool-call or barge-in interrupts the agent's current speech, not only at end_call. Branch P will not address this (Branch P only affects how the model sequences its goodbye turn). If post-Branch-P UAT shows these mid-call warnings still fire and correlate with caller-perceived mid-call cut-offs (the transcript shows two earlier truncations — `"Let me pull up the calendar for you real quick to"` and `"No problem at all. I'll get all that saved down"`), promote this to a Phase 60.3 follow-up or its own phase, not in-scope for Plan 3.

- **`playback_finished_at` stale-segment capture:** The current instrumentation captures whatever the `_GoodbyeDiagHandler` last saw on the framework logger. On a call with multiple SegmentSynchronizer warnings, only the LAST warning's timestamp wins, but that "last warning" may be an earlier segment's, not the final goodbye's, if the goodbye turn's playback_finished event never triggers the warning path (e.g., if it completes cleanly OR if session close fires before the warning). Instrumentation refinement (e.g., track per-segment, not latest-wins) is a candidate note for `60.3-SUMMARY.md` re-evaluation per D-A-03. Non-blocking for Plan 3.

- **Fix I (post-call pipeline 8s timeout) monitoring:** No `Post-call pipeline TIMEOUT after 8s` log observed on this call — passive monitoring per D-X-03 continues; no action required.

- **`tool_call_log_tail` empty despite `end_call` firing:** The diagnostic record's `tool_call_log_tail` came back as `[]` even though `end_call` clearly fired. This is by design (the current instrumentation does not log `end_call` into `tool_call_log_tail` — only other tools that run inside the agent's tool-call loop). Not a bug; note for future instrumentation refinement.
