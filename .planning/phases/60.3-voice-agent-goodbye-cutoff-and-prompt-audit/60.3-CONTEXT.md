## Implementation Decisions

### Stream A — Goodbye cut-off diagnosis + fix

- **D-A-01: Instrumentation logging surface — local logger + Sentry breadcrumbs.** Use `logger.info("[goodbye_race] %s", json.dumps({...}))` for forensic searching in Railway logs, AND `sentry_sdk.add_breadcrumb(category="goodbye_race", data={...})` so any goodbye-related Sentry event (participant-disconnect followed by `_SegmentSynchronizerImpl` warning, post-call pipeline error, etc.) carries the diagnostic context for the owner-side Sentry feed. **Reason:** owner already monitors Sentry; Railway log search is for forensic deep-dives. Both surfaces serve different operational roles. **Don't** write a Supabase `voice_diagnostic_events` row per call — DB writes on the hot path are out of proportion to a diagnostic-only feature.

- **D-A-02: Timestamps to capture — minimal set + audio-frame markers.** The diagnostic record per call must include:
  - `end_call_invoked_at` (when Gemini called the `end_call` tool — captured inside `create_end_call_tool` factory)
  - `last_text_token_at` (when the model emitted its last text delta before goodbye-segment close — hook into `RealtimeModel`/`AgentSession` text-stream event)
  - `last_audio_frame_at` (when the audio pipeline emitted its last frame for the goodbye segment — hook into LiveKit audio-publish path)
  - `playback_finished_at` (when `_SegmentSynchronizerImpl.playback_finished` was called, with `text_done` and `audio_done` flags captured)
  - `participant_disconnect_at` + `disconnect_reason` (CLIENT_INITIATED vs SERVER_INITIATED — already in agent.py logs, just attach to the diagnostic record)
  - `session_close_at`
  - **Reason:** without audio-frame timestamps we cannot distinguish "Gemini stopped emitting text" from "audio pipeline ate the last frame" — fewer UAT iterations to a real fix means the caller-facing bug ships sooner.
  - The exact hook points for `last_text_token_at`, `last_audio_frame_at`, `playback_finished_at` are RESEARCH items for Stream A Plan 1 — confirm against installed `livekit-agents 1.5.1` source before code is written.

- **D-A-03: Sample rate — every call for the duration of the phase, then re-evaluate.** Need a baseline of healthy goodbyes to compare against the broken one. ~50-100 KB extra logs per call is acceptable. **Re-evaluation trigger:** after the goodbye fix lands and N≥10 post-fix UAT calls show no recurrence, drop to race-detected-only sampling (only emit when `_SegmentSynchronizerImpl.playback_finished` warning fires with `text_done=false`). Re-evaluation captured as a follow-up note in `60.3-SUMMARY.md`, not a separate phase.

- **D-A-04: Capture transcript turns near goodbye.** Last 3 transcript turns (caller + agent) attached as a string field on the diagnostic record. **Reason:** without the actual goodbye text, the timestamps are uninterpretable — was Gemini's last token "Thank you for calling" (truncated immediately after the opener) or "Thank you for calling, have a great day, goodbye" (truncated at the very end)? Different races, different fixes.

- **D-A-05: Fix sequencing — iterate within phase.** Three-step plan shape: (1) ship instrumentation, (2) execute 1 UAT call, (3) analyze evidence and choose fix from data — either prompt-hardening of `_build_call_duration_section` (if evidence shows Gemini ignored the existing guidance) OR `end_call.py` tool-level guard (e.g., wait for `playback_finished` with `text_done=true` before scheduling `_delayed_disconnect`, if evidence shows audio-pipeline truncation). Then 2nd UAT to verify. **Reason:** ship a real fix this phase, not just instrumentation, but pick the fix from evidence not blind prompt hardening. The 12s `_delayed_disconnect` sleep already protects against agent-side disconnect, so the fix is almost certainly Gemini-side (prompt) or LiveKit-side (audio-pipeline guard) — both are knowable only with the instrumentation in place.

- **D-A-06: Cross-repo workflow unchanged from 60.2.** Stream A code lands in `C:\Users\leheh\.Projects\livekit-agent\` (Windows path; hyphen, not underscore — see 60.2-RESEARCH.md Pitfall 5). Commit with `--no-verify`. Push to Railway auto-deploy. Phase artifacts (PLANs, SUMMARY, AUDIT, UAT) live in this homeservice_agent directory.

### Stream B — Prompt best-practices audit

- **D-B-01: Deliverable shape — findings doc + per-section patch series.** `60.3-PROMPT-AUDIT.md` captures the audit findings with one section header per `_build_*_section` function, scored across the 7 dimensions in D-B-02. Each finding maps to a numbered patch plan (`60.3-NN-PLAN.md`) so individual changes are atomic, reviewable, revertable. **Reason:** a "big-bang" prompt rewrite makes UAT regressions impossible to triage and forces an all-or-nothing revert. Per-section atomicity keeps the owner from being whipsawed by a single bad change in a sea of good ones.

- **D-B-02: Audit dimensions (ranked by caller-UX impact).** Score each section against:
  1. **Anti-hallucination strength** — caller hearing wrong info (booking time, address, price) is the worst possible outcome
  2. **Realtime-model-specific phrasing** — Gemini Live ignores chat-LLM patterns; misalignment causes the model to "think out loud" or emit chat-style preambles
  3. **Section ordering** — Gemini attends more strongly to top of prompt; CRITICAL RULE-framed sections need top placement (already partially done in Phase 60)
  4. **STATE+DIRECTIVE consistency** — owner wants the AI to act decisively, not narrate tool-state to caller in plain English
  5. **VAD-redundant guidance** — "let caller finish" prose competes with the actual VAD config (Fix G `silence_duration_ms=1500`); confuses the model
  6. **Token economy** — shorter prompt = faster Gemini first-token = snappier conversation perceived by caller
  7. **Locale parity drift** — Spanish prompt path (`locale='es'`) must mirror English changes; tiered UX between en/es callers is a hidden owner-invisible bug

- **D-B-03: Audit scope — full 16 sections × 2 locales (en + es).** No subset cut. Spanish parity in same phase. **Reason:** half-audited prompt = half-improved caller UX. Hispanic homeowners are a large home-services demographic; deferring locale parity creates a tiered UX the owner can't see from the dashboard.

- **D-B-04: Section ordering for Stream B plans.** Highest-caller-impact-risk sections first, so if scope-cut becomes necessary the high-impact ones are already done:
  1. `_build_call_duration_section` (lines 544-556) — directly relates to the goodbye race; informed by Stream A evidence
  2. `_build_tool_narration_section` (lines 112-152) — already touched in 60 + 60.2 revert; verify final state is realtime-model-optimal
  3. `_build_voice_behavior_section` (lines 42-55) — sets the model's pacing, barge-in handling
  4. `_build_corrections_section` (lines 56-76) — caller misspeaks → model recovery; high anti-hallucination relevance
  5. `_build_outcome_words_section` (lines 77-111) — STATE+DIRECTIVE phrasing; common words the model speaks aloud
  6. `_build_info_gathering_section` (lines 304-396) — single-question framing locked in Phase 60; verify no drift
  7. `_build_booking_section` (lines 408-512) — booking confirmation, anti-hallucination on time/address readback
  8. Remaining 8 sections (identity, working_hours, greeting, language, repeat_caller, customer_account, intake_questions, decline_handling, transfer) — lower-risk batched in tail plans

- **D-B-05: Per-section patch criteria.** Each patch plan must (a) cite the audit dimension(s) the change addresses, (b) include the diff text inline in the plan, (c) include any test changes if `tests/test_prompt.py` assertions need updating, (d) be independently revertable (one git commit per plan).

- **D-B-06: Verification approach for Stream B.** UAT-driven, not unit-test-driven. Existing `tests/test_prompt.py` regression suite must stay green (101/101 from 60.2 baseline; new tests can be added per patch plan but the existing inverted assertions from 60.2 Plan 05 must remain). One end-to-end UAT call after Stream B Plan 1 (call_duration_section) and another after the final Stream B plan to confirm cumulative changes don't regress caller experience. No A/B comparison — single deployment per plan.
