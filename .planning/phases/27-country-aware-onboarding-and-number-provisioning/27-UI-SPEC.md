## Interaction Contracts

### Country Select Dropdown

- Render order: Singapore — United States — Canada
- On selection change: fire availability check API immediately (do not wait for form submit)
- While availability check is in flight: show `Loader2` spinner inline next to the availability badge (16px, `#475569`)
- After response: show availability badge below the select field
  - SG available (count > 0): `"{N} Singapore numbers available"` — text-xs, `#C2410C`
  - SG available = 0: trigger waitlist UI replacement (see below)
  - US / CA selected: no badge shown (dynamic provisioning, always available)

### Phone Number Field

- Rendered below full name field, above country select
- Auto-prefixes country code as placeholder hint based on current country selection:
  - Singapore: placeholder = `9123 4567` with prefix label `+65`
  - United States: placeholder = `555 000 0000` with prefix label `+1`
  - Canada: placeholder = `555 000 0000` with prefix label `+1`
- Implementation: display a static prefix inside the input row using a non-editable inline span styled to match input text (`text-base text-[#0F172A]`), user types only the local number
- On country change: clear phone field value, update prefix label

### Singapore Waitlist UI (zero availability)

When SG availability = 0, replace the normal step content below the heading with:
- Heading stays: "Your Details"
- Subheading changes to: "Singapore numbers are currently full. Join the waitlist and we'll notify you when a slot opens."
- Show only: email input + "Join waitlist" button
- Hide: full name, phone, country fields, and Continue button
- After successful waitlist join API call: replace form with success state (see Copywriting Contract)
- User cannot proceed through the wizard from this state — no Continue button

### Form Validation (client-side, before API call)

- Full name: required, minimum 2 characters
- Phone: required, minimum 7 digits after stripping non-numeric characters
- Country: required, must be one of `SG`, `US`, `CA`
- Inline errors appear below each field using `text-xs text-destructive` — not via Alert component
- Alert component is reserved for API-level errors only (network failures, server errors)

### Loading States

- Continue button: replace label with `Loader2` spinner (h-4 w-4 animate-spin) while API call is in flight
- Button disabled during loading
- Country availability check: inline spinner beside badge (does not disable Continue button)

## Screen States Summary

| State | Trigger | UI Shown |
|-------|---------|---------|
| Default | Step loads | Full name + phone + country dropdown + Continue |
| Country selected (US/CA) | User picks US or CA | Phone prefix updates, no badge, Continue enabled |
| Country selected (SG, slots > 0) | User picks SG, API returns count | Phone prefix updates, availability badge shown |
| SG availability checking | SG selected, API in flight | Spinner beside future badge position |
| SG full (0 slots) | API returns 0 | Waitlist UI replaces form content |
| Waitlist submit loading | User clicks "Join waitlist" | Button spinner, button disabled |
| Waitlist success | API confirms waitlist join | Success message, no further actions |
| Continue loading | User clicks Continue, API in flight | Button spinner, button disabled |
| API error | Any API failure | Alert (destructive) above form fields |
| Field validation error | User submits with invalid fields | Inline per-field error text (text-xs text-destructive) |
