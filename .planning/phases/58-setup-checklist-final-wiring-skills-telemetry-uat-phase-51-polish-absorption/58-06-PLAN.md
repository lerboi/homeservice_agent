## Architecture Overview

Two providers, one adapter contract:
- **Xero** (REST, `xero-node` SDK) — Phase 55
- **Jobber** (GraphQL, `graphql-request`) — Phase 56
- **Jobber schedule mirror** (local calendar_events) — Phase 57

Shared contract: `src/lib/integrations/` exposes
`getAuthUrl`, `exchangeCode`, `refreshTokenIfNeeded`, `fetchCustomerByPhone`,
`revoke` — each provider implements the same interface.

Database surface:
- `accounting_credentials` table (migration 052) — tokens, scopes, error_state,
  last_context_fetch_at, external_account_id, jobber_bookable_user_ids,
  jobber_last_schedule_poll_at.
- `calendar_events` (provider='jobber' since migration 055) — schedule mirror
  rows from Phase 57.

Data flow:
- **Dashboard reads** → Next.js 16 `'use cache'` + `cacheTag` via
  `src/lib/integrations/status.js`.
- **Call-path reads** → Python agent service-role direct → Xero/Jobber APIs,
  with 2.5s budget per Phase 55 D-04.
- **Webhook invalidation** → `/api/webhooks/{xero,jobber}` → HMAC verify →
  per-phone `revalidateTag('<provider>-context-${tenantId}-${E164}')`.

## Gotchas

1. **Next.js 16 forbids `'use cache'` on class methods** — use module-level
   cached functions (XeroAdapter.fetchCustomerByPhone is a module-level fn,
   not a class method).
2. **Cross-runtime casing divergence** is intentional — Next (camelCase)
   vs Python (snake_case). Don't "normalize" — you'll break one side.
3. **activity_log column names** are `event_type` + `metadata`, NOT `action`
   + `meta` despite what some historical CONTEXT docs say.
4. **Jobber webhook HMAC key = JOBBER_CLIENT_SECRET** (no separate
   JOBBER_WEBHOOK_SECRET env var — Pitfall 1 Option B from Phase 56).
5. **jobber_last_schedule_poll_at is SEPARATE from last_context_fetch_at** —
   schedule poll cursor, not customer-context touch. Don't conflate.
