After the existing call upsert in `processCallEnded`, add auto-cancel logic. The `metadata` parameter from Retell's `call_ended` event should contain the dynamic variables. Check `metadata?.test_call === 'true'` OR check if the `retell_metadata` stored in the calls row has `test_call`.

However, per Pitfall 2 from RESEARCH.md, Retell may not echo back `retell_llm_dynamic_variables` in `call_ended` payload. The SAFER approach: after upserting the call record, query it back and check `retell_metadata` — BUT `processCallEnded` writes `metadata` directly to `retell_metadata`. So if `metadata` from Retell contains `test_call`, it will be in `retell_metadata`.

**Fallback strategy if Retell does NOT include dynamic vars in call_ended metadata:** Use the approach from RESEARCH.md — store `call.call_id` returned by `createPhoneCall()` in the tenant row (e.g., `last_test_call_retell_id`). Then in `processCallEnded`, compare. BUT this requires a schema change. Simpler: just check `metadata?.test_call` first, and if the Retell payload structure doesn't include it, the executor should adjust at implementation time.

**Implementation — add after the existing `supabase.from('calls').upsert(...)` in processCallEnded:**

```js
// Auto-cancel test call bookings (D-08) — clean up calendar after onboarding test call
const isTestCall = metadata?.test_call === 'true' ||
                   metadata?.retell_llm_dynamic_variables?.test_call === 'true';

if (isTestCall && tenantId) {
  // Cancel any appointment created during this test call
  const { data: testAppt } = await supabase
    .from('appointments')
    .select('id')
    .eq('retell_call_id', call_id)
    .eq('tenant_id', tenantId)
    .maybeSingle();

  if (testAppt) {
    // Cancel the appointment (Pitfall 6: must also reset lead status)
    await supabase
      .from('appointments')
      .update({ status: 'cancelled' })
      .eq('id', testAppt.id);

    // Reset associated lead from 'booked' back to 'new'
    await supabase
      .from('leads')
      .update({ status: 'new', appointment_id: null })
      .eq('appointment_id', testAppt.id)
      .eq('tenant_id', tenantId);
  }
}
```

IMPORTANT: This logic runs in `processCallEnded` (fires immediately on call end per RESEARCH.md recommendation). Do NOT put it in `processCallAnalyzed` — that fires minutes later.

IMPORTANT: Auto-cancel must NOT fire during a live call — only post-call. The `processCallEnded` handler only runs on the `call_ended` event, which fires after the call disconnects. This is correct by design.
