**Required sections (in order):**

1. **Architecture Overview** — Table of components (slot calculator, booking engine, Google Calendar, Outlook Calendar, webhook handlers, cron jobs, API routes) + ASCII data flow diagram showing: tenant sets working hours -> calendar events sync from Google/Outlook -> slot calculator computes available slots -> AI offers slots -> atomicBookSlot locks slot -> calendar push via after()

2. **File Map** — Table with EVERY source file listed in read_first (except SKILL.md and STATE.md). Each row: file path, role description.

3. **Slot Calculator** deep dive — File path, `calculateAvailableSlots()` exact signature with ALL parameter names and types, algorithm walkthrough (walk forward from day open, skip overlaps/lunch/calendar blocks/travel buffers), travel buffer rules (no zones=30min, same zone=0, cross-zone=lookup or default 30min), pure function (no DB access).

4. **Atomic Booking** deep dive — File path, `atomicBookSlot()` signature, RPC `book_appointment_atomic` flow: advisory lock hash formula `pg_try_advisory_xact_lock(abs(hashtext(tenant_id || epoch(start_time))))`, tsrange overlap check, success/failure return shapes `{ success: true, appointment_id }` / `{ success: false, reason: "slot_taken" }`, secondary defense UNIQUE constraint.

5. **Google Calendar Integration** deep dive — File path, four exported functions with signatures: `pushBookingToCalendar()` (creates event, `[URGENT]` prefix for emergencies), `syncCalendarEvents()` (incremental sync via `last_sync_token`, full re-sync on 410), `registerWatch()` (push notification channel, 7-day TTL), `revokeAndDisconnect()` (cleanup). OAuth routes: auth initiation route, callback route (credential storage pattern).

6. **Outlook Calendar Integration** deep dive — File path, MSAL lazy singleton pattern, `graphFetch()` wrapper, token refresh via direct fetch (NOT MSAL cache — serverless-safe), deltaLink handling (store full URL as `last_sync_token`), exported functions: `pushBookingToCalendar`, `syncCalendarEvents`, `renewOutlookSubscription`. OAuth routes: auth initiation, callback (display name via direct Graph /me fetch).

7. **Webhook Handlers** deep dive — Google push notification handler (file path, how it triggers sync). Outlook change notification handler (file path, validation token handling).

8. **Cron Jobs** — `renew-calendar-channels` route: dual-provider channel renewal logic, how it handles Google watch renewal + Outlook subscription renewal.

9. **API Routes** — appointments GET/POST, appointments/[id] PATCH/DELETE, working-hours GET/PATCH, zones route. For each: HTTP methods, key query params, response shapes.

10. **Database Tables** — Table format. From 003_scheduling.sql: appointments (UNIQUE constraint, tenant_id FK, external_event_id, external_event_provider from 007), service_zones, zone_travel_buffers, calendar_credentials (is_primary from 007), calendar_events. Include key columns for each.

11. **Environment Variables** — Table: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, MICROSOFT_CLIENT_ID, MICROSOFT_CLIENT_SECRET, MICROSOFT_TENANT_ID, NEXT_PUBLIC_APP_URL (callback URLs), CRON_SECRET.

12. **Key Design Decisions** — Bulleted list, each with WHY not just WHAT. Mine from STATE.md decisions tagged with Phase 03, Phase 08. Must include:
    - Local DB mirror is source of truth (never live-query during call)
    - pg_try_advisory_xact_lock non-blocking (prevents queue buildup)
    - UNIQUE (tenant_id, start_time) as secondary defense
    - Travel buffer logic rules
    - after() for calendar push (never in synchronous call path)
    - Store full deltaLink URL as last_sync_token
    - Direct fetch for Outlook token refresh (serverless-safe)
    - is_primary flag for multi-provider calendar push
    - PROVIDER_CONFIG map centralizes auth endpoints per provider
    - Optimistic UI for make-primary badge swap
