## Color

60/30/10 allocation matches existing dashboard pages — no new color tokens introduced.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-background` → `--background` (white in light, near-black in dark) | Page background via dashboard layout |
| Secondary (30%) | `bg-card` → `--card` | Provider card surface (`card.base` token) |
| Accent (10%) | `bg-[var(--brand-accent)]` → `#C2410C` light / `#FB923C` dark | **Primary "Connect Xero" / "Connect Jobber" button ONLY** |
| Destructive | `border-red-600 text-red-600 hover:bg-red-50` (light) + equivalent dark variants | Disconnect button outline + confirm dialog action |

**Accent reserved EXCLUSIVELY for:**

1. The "Connect Xero" button (when Xero disconnected)
2. The "Connect Jobber" button (when Jobber disconnected)

NOT used for: text links, section headings, icons, connected-state indicators, disconnect buttons, status pills, or focus rings (focus rings use the same accent via `focus.ring` token — that IS on the approved reserved list because it's the focus ring convention, not a standalone decorative accent).

**State-specific colors:**

| State | Visual |
|-------|--------|
| Disconnected (idle) | Card default surface, muted foreground status text, accent-colored "Connect X" button |
| Connected (success) | Card default surface, `text-emerald-600` status text inline dot/check, outline "Disconnect" button |
| Connecting (OAuth redirecting) | Button shows `Loader2` spinner, text swaps to "Connecting…", disabled |
| Disconnecting | Button shows `Loader2` spinner, text swaps to "Disconnecting…", disabled |
| Error (OAuth callback returned error param) | `toast.error(...)` via sonner — no in-card error state; card reverts to disconnected |
| Loading (initial fetch) | Skeleton card replicating card shape (icon placeholder + name placeholder + status placeholder + button placeholder) |

Emerald green (`text-emerald-600` / `dark:text-emerald-400`) for connected state is re-used from the existing integrations page — not a new accent, serves a semantic "success" role and never competes with `--brand-accent`.


1. Page fetches `GET /api/integrations/status` on mount (Phase 54 new endpoint)
2. Page reads `invoicing` flag from `FeatureFlagsProvider` context (Phase 53 existing)
3. Status + flag combine to pick correct status-line string (one of the three variants above)

5. OAuth callback lands at `/dashboard/more/integrations?connected=xero` → page reads `searchParams` → fires `toast.success`
6. Disconnect confirm → `POST /api/integrations/disconnect` with `{ provider }` body → on success, optimistically clear from local state

## Interaction Contracts

### OAuth Initiation

1. Owner clicks "Connect Xero" → button disables, shows `Loader2` + "Connecting…"
2. Frontend calls `GET /api/integrations/xero/auth` — returns `{ url: "https://login.xero.com/..." }`
3. Frontend sets `window.location.href = url` — **full-page redirect**, NOT `router.push()` and NOT popup
4. Browser navigates to provider OAuth consent page
5. On approval, provider redirects to `/api/integrations/xero/callback?code=...&state=...`
6. Callback route handler exchanges code, writes `accounting_credentials` row, calls `revalidateTag(\`integration-status-${tenantId}\`)`, redirects to `/dashboard/more/integrations?connected=xero`
7. Page re-renders with connected status; `useEffect` on `searchParams` fires `toast.success("Xero connected.")`

**Error path:** If API returns error or network fails before redirect, button re-enables with original label, `toast.error` fires.

### Disconnect

1. Owner clicks "Disconnect" button on connected card → opens `AlertDialog` with destructive confirmation
2. Owner clicks "Disconnect" (red) in dialog → dialog closes, button enters disconnecting state (`Loader2` + "Disconnecting…")
3. Frontend calls `POST /api/integrations/disconnect` with `{ provider: 'xero' }`
4. On success, local state removes the connection → card re-renders in disconnected state (no toast; visual change is self-evident)
5. On error, `toast.error`, state reverts

**Cancel path:** Dialog close or "Cancel" button → no API call, no state change.

### Loading (initial page render)

- Render Skeleton versions of both provider cards side-by-side in the grid
- Skeleton replicates: `h-8 w-8 rounded-lg` icon, `h-4 w-24` name, `h-4 w-full` status line (two lines), `h-9 w-full rounded-md` button
- Skeleton uses `animate-pulse` default from shadcn `Skeleton`
- Once `GET /api/integrations/status` resolves, skeletons swap to real cards (no fade — immediate replace is fine; matches existing page)

### Keyboard

- All buttons tabbable in natural order: Xero Connect/Disconnect → Jobber Connect/Disconnect
- `focus-visible` rings use `focus.ring` token from design-tokens.js (`focus:ring-2 focus:ring-[var(--brand-accent)] focus:ring-offset-1`)
- `AlertDialog` traps focus per Radix default; Escape closes; first focus lands on Cancel (not Disconnect — safer default for destructive action)

### Responsive

- At `<768px` (default mobile): grid is single-column — both cards stack full-width
- At `>=768px` (md breakpoint): `grid-cols-2` — Xero and Jobber side-by-side
- Card internal layout does not change across breakpoints — stays flex-column with icon row on top, status in middle, button at bottom

## Dark Mode

All color tokens used in this phase resolve through CSS variables (`--background`, `--card`, `--muted-foreground`, `--border`, `--brand-accent`, `--destructive`) which already have dark variants declared in `src/app/globals.css`. No additional `dark:` prefixes needed for the primary surfaces.

**Two explicit `dark:` prefixes required:**

1. Connected-state status text: `text-emerald-600 dark:text-emerald-400` (emerald is not in the CSS variable palette)
2. Disconnect button hover: `hover:bg-red-50 dark:hover:bg-red-950/40` (red-50 is too bright in dark mode)

All other elements inherit theme automatically. Phase 49 dark-mode work (DARK-03) already migrated dashboard components to CSS-variable-backed classes — Phase 54 continues that discipline.

## Accessibility

- All icons decorative: wrapped with `aria-hidden="true"` on lucide components; provider name carried in visible text
- Status lines use semantic `<p>` — no `role="status"` needed (not a live region; status updates are triggered by page actions, not background events)
- Destructive confirmation: `AlertDialog` provides ARIA live-region semantics by default via Radix
- Loading skeletons: `Skeleton` component is visual-only; consider adding `aria-busy="true"` on the grid container during initial load
- Color is never the sole state signal — connected state uses both emerald text AND the "Connected." prefix in the status line; button state uses both spinner icon AND text change ("Connecting…")
