## Phase Boundary

Three narrow, mostly-orthogonal reliability-bug streams, all landing in `C:\Users\leheh\.Projects\livekit-agent\` (cross-repo — same workflow as Phases 60.2 / 60.3):

**Stream A — Booking timezone hardening (customer-facing).** The +8h offset observed in live traffic (caller asked for 3 PM SG, Google Calendar event landed at 11 PM) is intermittent — mostly correct, sometimes wrong. Inline audit during discuss pinpointed the mechanism: `google_calendar.create_calendar_event()` (`livekit-agent/src/lib/google_calendar.py:101-106`) builds the event body with `{"dateTime": iso}` only — **no `timeZone` field**. Combined with the documented Gemini behaviour at `livekit-agent/src/tools/book_appointment.py:55-72` ("Gemini occasionally strips the '+00:00' offset when re-emitting slot_start from check_availability"), the intermittency pattern is: when Gemini preserves the offset, Google respects it → correct time; when Gemini strips the offset, a naive `dateTime` is sent without `timeZone` override → Google interprets it in the calendar's default timezone (not the tenant's) → arbitrary shift that happens to match UTC+8 for SG tenants. The `_ensure_utc_iso` defensive helper is only called on the **SMS formatting** path, not before DB persistence or before `create_calendar_event()`. Harden every datetime boundary.

**Stream B — STT language pinning.** Pin Gemini Live's input language on session start to stop the German/Hindi hallucinations emitted during silence / low-SNR segments. Touch point: `livekit-agent/src/agent.py` where `RealtimeModel` is configured. Exact kwarg name, allowed values, and list-mode behaviour on the pinned stack (`livekit-plugins-google@43d3734` + Gemini 3.1 Flash Live) are research items — confirm in RESEARCH before planning.

**Stream C — 16s `end_call` stall diagnosis + fix (folded from 60.3-SUMMARY follow-up).** After 60.3 Branch P shipped, post-fix UAT shows the goodbye message now completes fully but Gemini waits ~16s before invoking the `end_call` tool (see `60.3-HUMAN-UAT.md:150`). This is a different race class from the 60.3 goodbye-truncation race (which was audio-pipeline-side); the 16s gap is `last_text_token_at → end_call_invoked_at`, i.e., Gemini-side tool-invocation latency, not audio pipeline. Mirror 60.3 Stream A diagnose-first pattern: reuse the `[goodbye_race]` instrumentation already in place, execute one UAT, evidence-pick the fix.

**Out of scope (carried from 60.2 / 60.3 locks):** No DB migrations. No tool signature changes. No `livekit-agents` upgrade past 1.5.1. No google-plugin upgrade past commit `43d3734`. No STATE+DIRECTIVE tool-return contract changes (Phase 60). No prompt.py edits unless Stream C evidence points there. No Next.js / dashboard work. No `_delayed_disconnect` 12s sleep tuning without instrumented justification.

## Implementation Decisions

### Stream A — Booking timezone hardening

- **D-A-01: Pass `timeZone` in `events.insert()` body — structural fix.** In `livekit-agent/src/lib/google_calendar.py` `create_calendar_event()`, change the `start` / `end` dicts to include the tenant's IANA timezone:
  ```python
  "start": {"dateTime": appointment["start_time"], "timeZone": tenant_timezone},
  "end":   {"dateTime": appointment["end_time"],   "timeZone": tenant_timezone},
  ```
  Per Google Calendar API: when `timeZone` is present, it is authoritative — Google interprets `dateTime` in that zone even if the string is naive, eliminating the "calendar default TZ" failure mode. Requires plumbing `tenant_timezone` into `create_calendar_event()` (currently doesn't receive it). **Reason:** structural, not defensive — closes the root-cause hole identified during discuss audit.

- **D-A-02: Plumb `tenant_timezone` through to `create_calendar_event()`.** Caller is `book_appointment` tool (and any other call sites — verify in RESEARCH). Tenant timezone is already fetched in `book_appointment` (`book_appointment.py:266`, `tenant_timezone = (tenant.get("tenant_timezone") if tenant else None) or "America/Chicago"`). Pass the resolved value through. **Reason:** D-A-01 requires it; no new DB read needed.

- **D-A-03: Apply `_ensure_utc_iso` at every persistence and push boundary — defensive.** Currently only called on SMS formatting paths (`_format_date_for_sms`, `_format_time_for_sms`). Extend to:
  - Before writing `slot_start` / `slot_end` to the `appointments` row
  - Before calling `create_calendar_event()`
  - Before any `activity_log` write that persists a slot ISO
  - (Verify exhaustive boundary list during RESEARCH + PLAN-1 audit)
  **Reason:** even with D-A-01's `timeZone` override, we still want DB rows to carry offset-suffixed ISOs so downstream consumers (dashboard, Xero, Jobber, SMS recap) read unambiguous values.

- **D-A-04: Naive-ISO handling — coerce-and-log, never raise.** When `_ensure_utc_iso` encounters `dt.tzinfo is None`, preserve the current behaviour (re-attach `+00:00`, contract: `check_availability` returns UTC) AND emit a structured WARN log: `logger.warning("[tz_coerce] naive ISO at boundary=%s call_id=%s iso=%s", boundary, call_id, iso_str)`. Do not add a fail-loud `_assert_utc_iso` variant. **Reason:** a mid-call `raise` fails the booking in front of a live caller — worst-possible UX for a corner case we are explicitly defending against. Logs surface regressions for follow-up without crashing production.

- **D-A-05: Null `tenant_timezone` fallback — UTC + WARN, not `America/Chicago`.** Change all THREE fallback sites (`book_appointment.py:266`, `check_availability.py:142`, AND `post_call.py:612` — third site surfaced by 60.4 RESEARCH, 2026-04-23) from `... or "America/Chicago"` to `... or "UTC"` plus a `logger.warning("[tenant_config] null tenant_timezone tenant_id=%s — falling back to UTC", tenant_id)`. **Reason:** (1) Chicago is a legacy US-only default that silently produces wrong times for every non-US tenant with a null column (SG, CA, UK); (2) UTC is a neutral default — a wrong UTC offset is visibly wrong and surfaces quickly, whereas a wrong Chicago offset (UTC-5/-6) is close enough to pass unnoticed for some tenants; (3) aligns with Phase 61/62 multi-tenant direction. RESEARCH item: confirm how many prod tenants currently have `tenant_timezone = NULL` (spot-check Supabase); if zero, this is a pure forward-safety change.

- **D-A-06: Regression test — unit + one UAT call.** Unit test(s) in the livekit-agent repo covering:
  - Naive ISO fed to `_ensure_utc_iso` emerges suffixed with `+00:00` and a WARN is logged
  - `create_calendar_event` event body carries `timeZone` matching the passed `tenant_timezone` in both `start` and `end`
  - Null tenant_timezone → UTC fallback + WARN (both `book_appointment` and `check_availability`)
  One UAT call on `+6587528516` (SG tenant, the 2026-04-21 repro case): book a 3 PM SG slot, assert Google Calendar event lands at 3 PM SG (i.e., 07:00 UTC) — not 11 PM. **Reason:** unit tests catch the offset-stripping corner case without needing to reproduce it via a live call; UAT confirms end-to-end including Gemini's variable offset behaviour.

### Stream B — STT language pinning

- **D-B-01: Source of truth — research-then-decide (lean toward per-tenant `default_locale`).** gsd-phase-researcher must confirm, before planning locks:
  1. The exact kwarg on `livekit-plugins-google@43d3734` + Gemini 3.1 Flash Live (is it `language_code="en-US"`? `languages=["en-US"]`? `language="en"`? undocumented?)
  2. Allowed value format (BCP-47 `en-US` vs ISO `en` vs Google-specific)
  3. List-mode behaviour if supported — does a list pin both and reject others, or does it fall back to "auto" (which would re-admit the hallucination)?
  4. Interaction with `RealtimeModel` instantiation timing (session-start vs per-turn)
  **Provisional lean:** per-tenant, read `tenants.default_locale` (already used by Phase 60.3 for prompt locale selection) → map `en` → `en-US`, `es` → `es-US`, pass to `RealtimeModel(<kwarg>=...)` at session start. Single source of truth for prompt locale AND STT locale. Lock the concrete approach in PLAN-B-1 after research.

- **D-B-02: Regression test — deliberate-silence UAT (softened acceptance, 2026-04-23).** One UAT call: (a) speak a short English utterance, (b) go silent for 10-15s (mic hot), (c) speak again. Assert **non-English token rate in the silence window is measurably reduced vs the pre-fix baseline** — NOT zero. 100% elimination is not achievable on the pinned native-audio model per Google docs (native-audio auto-detects language; `language_code` is defense-in-depth, not authoritative). Bonus if Railway log shows the `language` kwarg on the RealtimeModel instantiation log line AND the prompt directive appears in the system message. **Reason:** softened per RESEARCH finding (Google Live API docs + LiveKit community report confirm `language_code` is explicitly unsupported on native-audio preview models).

- **D-B-03: RE-OPENED 2026-04-23 — prompt directive + kwarg, belt-and-suspenders.** Stream B now has TWO surfaces:
  1. **SDK layer** — pass `language="en-US"` (or per-tenant-locale mapping) to `RealtimeModel(...)` in `agent.py`. Defense-in-depth; may be ignored by native-audio but costs nothing.
  2. **Prompt layer** — add a short directive to the system prompt pinning the caller-facing language. Location: Claude's discretion (likely `_build_voice_behavior_section` or a new `_build_language_section` in `prompt.py`). Exact wording: Claude's discretion, must respect Phase 60 STATE+DIRECTIVE contract and 60.3 locale parity. Per-tenant locale via `tenants.default_locale` (single source of truth for prompt locale + STT locale pin).

  **Reason:** RESEARCH confirmed the pinned Gemini native-audio model ignores `language_code`; the only documented community workaround for hallucinations is a system-prompt directive. Original D-B-03 "no prompt changes" rationale (orthogonal surface) no longer holds — the ASR hallucination can only be influenced from the prompt on this model. 60.3 Stream B scope was prompt audit, not language pinning — no overlap.
