Railway log showed `[tz_coerce] naive ISO at boundary input='2026-04-27T14:00:00'` — Gemini passed `14:00:00` naive (caller's SGT wall-clock digits), not the authoritative `06:00:00+00:00` (14:00 SGT = 06:00 UTC). `_ensure_utc_iso` coerced the naive ISO to UTC, shifting the booking by 8h.

- **Structural fix shipped:** commit `5e48273` replaces "pass slot_start_utc VERBATIM" prompt directive with an opaque `slot_token` that `check_availability` stashes on `deps["_slot_tokens"]` and `book_appointment` resolves server-side. ## Out-of-scope fixes shipped during pause

Both commits are on `livekit-agent/main` and deployed to Railway. They are genuinely structural (not hacks) and have full test coverage:

### Fix 1 — `b46851b` — slot cache prefetch

- **Problem:** `check_availability` ran 5 parallel Supabase queries every call (~500 ms). During that 500 ms window, any caller barge-in triggered Gemini's server-side `toolCallCancellation`, which `livekit-plugins-google` silently logs and drops. Agent then loops on filler phrases with no tool result.
- **Fix:** Extended `_run_db_queries` in `src/agent.py` to prefetch `appointments`/`calendar_events`/`service_zones`/`zone_travel_buffers`/`calendar_blocks` in the same parallel gather. Cache stashed on `deps["_slot_cache"]` with `fetched_at` timestamp. `check_availability` consumes cache when TTL < 30 s; repopulates on live-fetch miss. `book_appointment` invalidates cache on successful booking.
- **Effect:** First `check_availability` per call drops from ~500 ms → ~50 ms. Barge-in cancellation window shrinks ~10×.
- **Files:** `src/agent.py`, `src/tools/check_availability.py`, `src/tools/book_appointment.py` (cache invalidation).
- **Tests:** `tests/test_check_availability_slot_cache.py` (3 tests). Includes `_TrippedSupabase` stub that raises if scheduling tables are queried despite a fresh cache — proves bypass.

### Fix 2 — `5e48273` — slot_token structural handoff

- **Problem:** Gemini 3.1 Flash Live ignores "pass slot_start_utc VERBATIM" directives. Rebuilds naive ISO from caller speech. Plan 01's `_ensure_utc_iso` coerces naive → UTC, producing 8h-off bookings for SG tenants (confirmed in UAT #2).
- **Fix:** `check_availability` generates `slot_token = "slot_" + secrets.token_hex(4)`, maps it to (slot_start_utc, slot_end_utc) on `deps["_slot_tokens"]` with 10-min TTL. STATE line now carries `slot_token=slot_<hex>` alongside the raw UTC values. DIRECTIVE updated to explicitly instruct Gemini to echo slot_token, citing the "off-by-8-hours" failure. `book_appointment` accepts `slot_token` as first param, resolves to authoritative UTC server-side, ignores any Gemini-constructed `slot_start`/`slot_end` when token is valid. Legacy path preserved for no-token / expired-token cases.
- **Tool description** rewritten to lead with "CRITICAL: pass slot_token" + ground the rule in the concrete past failure.
- **Effect:** Gemini is no longer in the critical path for constructing ISO strings. Authoritative UTC flows end-to-end via opaque token.
- **Files:** `src/tools/check_availability.py` (register + STATE line), `src/tools/book_appointment.py` (resolve + description + signature — `slot_token` is the new first positional arg; all other args have defaults for backward compat).
- **Tests:** `tests/test_slot_token_handoff.py` (9 tests). Primary test: `test_valid_token_overrides_gemini_hallucinated_iso` explicitly simulates Gemini passing `2026-04-27T14:00:00` (wrong) while supplying the correct token — asserts authoritative `06:00:00+00:00` is used.
