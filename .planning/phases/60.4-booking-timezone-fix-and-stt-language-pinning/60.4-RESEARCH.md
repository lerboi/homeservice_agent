## Architecture Patterns

### Google Calendar `events.insert()` timezone semantics — the authoritative source

Per Google's official API reference (quoted verbatim):

> **`start.dateTime`:** "The time, as a combined date-time value (formatted according to RFC3339). A time zone offset is required **unless** a time zone is explicitly specified in `timeZone`."
>
> **`start.timeZone`:** "The time zone in which the time is specified. (Formatted as an IANA Time Zone Database name, e.g. 'Europe/Zurich'.) For recurring events this field is required and specifies the time zone in which the recurrence is expanded."

[CITED: developers.google.com/workspace/calendar/api/v3/reference/events/insert]

**Implication:** offset-suffix AND `timeZone` are explicitly documented as **alternatives**, not concurrent. Specifying both has no documented precedence rule — community reports confirm the API applies the offset first, then the timeZone, yielding **double-offset shifts**. [CITED: community.n8n.io/t/timezone-issue-when-creating-google-calendar-events/92828 + multiple copy-programming / GitHub issue reports]

### Pattern: the correct Stream A body shape

**Source: `livekit-agent/src/lib/outlook_calendar.py:113` — already in this repo**

```python
# outlook_calendar.py — existing, working precedent for the Stream A fix shape
"start": {"dateTime": appt["start_time"], "timeZone": tz},
"end":   {"dateTime": appt["end_time"],   "timeZone": tz},
```

**Where `appt["start_time"]` is a UTC-offset-suffixed ISO (e.g., `"2026-04-30T06:00:00+00:00"`).** The Outlook API's docs and runtime behavior accept offset + timeZone and behave as expected — but **Google Calendar does not** (double-offset). So the Google fix cannot be a direct copy.

### Pattern: correct shape for Google specifically (what PLAN-A-1 must implement)

Two viable shapes — both correct per Google docs; pick one and stay consistent:

**Option 1 — naive local ISO + timeZone (recommended):**

```python
# Convert UTC-offset ISO to naive local ISO in the tenant timezone
from datetime import datetime
from zoneinfo import ZoneInfo

def _to_naive_local_iso(utc_iso: str, tenant_timezone: str) -> str:
    """Convert '2026-04-30T06:00:00+00:00' in UTC to '2026-04-30T14:00:00' in SGT."""
    dt = datetime.fromisoformat(utc_iso).astimezone(ZoneInfo(tenant_timezone))
    return dt.replace(tzinfo=None).isoformat()

event_body = {
    "summary": summary,
    "description": description,
    "start": {
        "dateTime": _to_naive_local_iso(appointment["start_time"], tenant_timezone),
        "timeZone": tenant_timezone,
    },
    "end": {
        "dateTime": _to_naive_local_iso(appointment["end_time"], tenant_timezone),
        "timeZone": tenant_timezone,
    },
}
```

**Option 2 — UTC-offset ISO, no `timeZone` field (current behavior):**

```python
# What's there today
event_body = {
    "start": {"dateTime": appointment["start_time"]},  # offset-suffixed
    "end":   {"dateTime": appointment["end_time"]},
}
```

Option 2 is **what already exists** and it works **when `appointment["start_time"]` carries a real UTC offset**. The bug is that Gemini sometimes strips the offset → `start_time` becomes naive → Google interprets it in the calendar's default TZ (which may not match the tenant). Option 1 closes this hole **even when `_ensure_utc_iso` fails**, because the event-body `timeZone` is authoritative. It's the belt-and-suspenders fix the CONTEXT calls out.

**PLAN-A-1 must NOT implement the naive "pass both offset-suffixed dateTime AND timeZone" variant** — that is the double-offset trap. Add this as a Pitfall in the plan.

### Anti-Patterns to Avoid

- **Mixing offset-suffixed `dateTime` and `timeZone` on Google Calendar.** Double-offset. Pick one channel; for the Stream A fix, pick Option 1 above.
- **Assuming `language="en-US"` pins the input language on a native-audio Gemini model.** It doesn't (see §Stream B Native-Audio Blocker). Ship it as defense-in-depth, not as a fix.
- **Re-introducing `session.say()` on `AgentSession(llm=RealtimeModel)`-only sessions.** 60.2 Fix H reverted this (runtime error: no TTS attached). Stream C's SDK-timer fallback candidate is NOT `session.say()` — it would have to be a direct `ctx.shutdown()` trigger on a post-farewell timer or a forced-tool-invocation via `session.generate_reply(...)`.
- **Raising exceptions in `_ensure_utc_iso` or its wrapper.** D-A-04 explicit: log + continue. A `raise` mid-call crashes the booking in front of a live caller.
- **Changing tool signatures.** D-X-04 locks. Stream B must not add kwargs to `book_appointment` etc.

## Stream A — Google API Semantics (details)

### Assertion 1: `timeZone` on `events.insert()` is authoritative when `dateTime` is naive

- **Status:** VERIFIED HIGH
- **Evidence:** Google's official reference docs quote (in §Architecture Patterns above): *"A time zone offset is required unless a time zone is explicitly specified in `timeZone`."*
- **Behavior under Option 1 (naive local ISO + `timeZone`):** Google interprets `dateTime` in the specified IANA zone → event lands at the intended wall-clock time → calendar default TZ is irrelevant. **This is the structural fix D-A-01 prescribes.**

### Assertion 2: Mixing offset-suffixed `dateTime` + `timeZone` produces double-offset errors

- **Status:** MEDIUM — behavior is documented as undefined by Google, confirmed by multiple independent community reports but not by Google's authoritative docs
- **Evidence:**
  - [CITED: community.n8n.io/t/timezone-issue-when-creating-google-calendar-events/92828] Users observed events placed at wrong times when both fields were set.
  - [CITED: github.com/googleapis/google-api-php-client/issues/957] "Google Calendar Api is setting wrong time on an event."
  - [CITED: Multiple blog posts / copyprogramming.com howto] "The most common cause of errors is accidentally specifying both..."
- **Implication for PLAN-A-1:** the fix MUST be Option 1 (naive local ISO + `timeZone`) — NOT the naive shape of "pass the offset-suffixed ISO and also add `timeZone`". Lock this as a Pitfall in the plan.

### Assertion 3: `outlook_calendar.py` uses offset-suffixed + `timeZone` without issues

- **Status:** VERIFIED HIGH (for Microsoft Graph / Outlook — NOT Google)
- **Evidence:** `lib/outlook_calendar.py:113` — this is the in-repo precedent that motivated the CONTEXT D-A-01 direction.
- **Caveat:** Outlook's API has different semantics from Google's. The Outlook shape does NOT transfer to Google (→ double-offset). Do not assume feature parity. **This is a subtle trap** — the CONTEXT's example code block (`"start": {"dateTime": appointment["start_time"], "timeZone": tenant_timezone}`) is correct for Outlook but would double-offset for Google. ## Common Pitfalls

### Pitfall 1: Double-offsetting on Google Calendar events

- **What goes wrong:** Passing both offset-suffixed `dateTime` AND `timeZone` to `events.insert()` shifts the event by (calendar-default TZ offset × 2) or by (offset + timeZone offset).
- **Why it happens:** Copying the Outlook shape verbatim (since `outlook_calendar.py` already has `{"dateTime": utc_iso, "timeZone": tz}` and works). Outlook and Google handle this differently.
- **How to avoid:** Strip the offset before passing `timeZone`. Use Option 1 pattern (naive local ISO + timeZone) in `google_calendar.py`.
- **Warning signs:** Event lands at (correct_time ± offset × 2). Unit test with a non-UTC tenant_timezone (SG = UTC+8 is perfect — divergent from the Railway container's UTC default) will catch this instantly.

### Pitfall 2: Assuming `language="en-US"` pins input on native-audio Gemini

- **What goes wrong:** Planning treats Stream B as a definitive fix; UAT fails; scope balloons.
- **Why it happens:** Intuitive reading of the kwarg name + prior experience with non-realtime STT pinning.
- **How to avoid:** Accept the native-audio blocker upfront; frame Stream B as defense-in-depth + prompt directive; set a softened UAT acceptance criterion.
- **Warning signs:** PLAN-B success criteria that say "transcript contains only English tokens" (too strict). Replace with "defense-in-depth applied; rate of non-English tokens measurably reduced; prompt directive engaged as documented community mitigation."

### Pitfall 3: Writing `_ensure_utc_iso` into `google_calendar.py` instead of fixing the body shape

- **What goes wrong:** D-A-03's "apply `_ensure_utc_iso` at every boundary" wording could lead to `_ensure_utc_iso(appointment["start_time"])` being called inside `google_calendar.py` — but that produces an offset-suffixed ISO, and if the `timeZone` field is also set (D-A-01), we've re-created Pitfall 1.
- **Why it happens:** Literal reading of D-A-03.
- **How to avoid:** The correct D-A-03 implementation for the Google path is Option 1 (convert to naive local ISO). The defensive `_ensure_utc_iso` is already on the tool entry path (site #1); site #4 (google_calendar) needs the structural fix, not a redundant sanitizer.
- **Warning signs:** PR diff that adds `_ensure_utc_iso(appointment["start_time"])` AND adds `timeZone` to the event body simultaneously.

### Pitfall 4: `A2A_ONLY_MODELS` branch — the reason the plugin is pinned

- **What goes wrong:** Any attempt to upgrade `livekit-plugins-google` past `43d3734` breaks `session.generate_reply()` on `gemini-3.1-flash-live-preview` (the greeting-injection path).
- **Why it happens:** Later plugin commits refactored the model-class detection branch and `gemini-3.1-flash-live-preview` either falls out of the allowlist or the reply-injection path errors.
- **How to avoid:** D-X-04 is non-negotiable. If Stream B blocker forces a model switch, it's a Phase 60.5+ scope — NOT in 60.4.
- **Warning signs:** PLAN-B task that says "update pyproject.toml plugin pin." Reject at PLAN-CHECK.

### Pitfall 5: `session.say()` on RealtimeModel-only session (60.2 Fix H revert)

- **What goes wrong:** Runtime error: `trying to generate speech from text without a TTS model`.
- **Why it happens:** `AgentSession(llm=RealtimeModel)` has no separate TTS — Gemini emits audio directly.
- **How to avoid:** Stream C's force-timer fallback cannot use `session.say()`. Must use `ctx.shutdown()` or `session.generate_reply(instructions="...")`.
- **Warning signs:** Reappearance of `session.say(` in `end_call.py` or `agent.py`.

from datetime import datetime
from zoneinfo import ZoneInfo

def _to_naive_local_iso(utc_iso: str, tenant_timezone: str) -> str:
    """

    return dt.replace(tzinfo=None).isoformat()
