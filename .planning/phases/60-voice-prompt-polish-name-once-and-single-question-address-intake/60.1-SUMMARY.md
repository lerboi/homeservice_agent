# Phase 60.1: Voice Prompt Polish — Live Call Regression Fixes

**Hotfix for three issues surfaced in the first live UAT call after Phase 60 shipped: caller SMS showing the wrong time, tool-call voice glitches from server-VAD cancellations, and redundant over-confirmation before tool calls. (A fourth issue — ~4s startup silence — was deferred at user request.)**

## Performance

- **Duration:** ~45 min (diagnostic + fix + commit)
- **Completed:** 2026-04-19
- **Commits:** 2 in livekit_agent, pushed to `main` (`e6b8828..0891241`)

## Accomplishments

### Issue 2 — Caller SMS wrong time (Phase 60 regression)

Root cause: Phase 60's `check_availability` STATE line introduced a human-readable `speech=` field alongside the UTC ISO `start=` field. Gemini sometimes synthesized `slot_start` for `book_appointment` from the `speech=` form, dropping the `+00:00` offset. The resulting naive datetime flowed into `_format_time_for_sms` where Python's `.astimezone()` silently assumed system-local tz (UTC on Railway) → wrong wall-clock time in SMS.

**Layer 1 fix (defensive parsers):**
- New `_ensure_utc_iso()` helper in `book_appointment.py` (lines 52-72). Canonicalizes any ISO form to UTC; treats naive as UTC (never falls through to system-local).
- New `_coerce_utc_aware()` helper in `utils.py` (shared across `format_slot_for_speech` and `to_local_date_string`).
- `book_appointment` canonicalizes `slot_start` and `slot_end` at function entry so every downstream consumer (SMS, calendar RPC, idempotency cache, deps stamping, readback) sees UTC ISO.
- Returns `STATE:booking_invalid reason=malformed_slot_iso` if parsing fails — never silently degrades to wrong-timezone.

**Layer 2 fix (unambiguous tool-param contract):**
- Renamed STATE fields: `start=` / `end=` → `slot_start_utc=` / `slot_end_utc=`. The `_utc` suffix + parameter-name parity makes Gemini's pass-through semantics unambiguous.
- DIRECTIVE now says: "pass slot_start_utc VERBATIM as slot_start ... the +00:00 offset MUST be preserved. Do not reformat, convert, or rebuild from the speech string." Applied to both `slot_available` and `slot_not_available` (alternatives) branches.

### Issue 3 — Voice glitches during tool calls

Root cause: **4 `server cancelled tool calls` warnings** in a single 237s UAT call, co-occurring with choppy transcript ("Just to confirm... [cut]", "\n\n My check shows..."). Three compounding causes:

**3A — AI over-confirms before tool calls.** Transcript pattern: `user: "9am"` → `agent: "Just to confirm, you're asking about 9 AM tomorrow, which is Monday, April 20th? Is that correct?"` → wait for "Yes" → filler → tool call. ~8s of redundant talk per slot check. Fix: added "NO TIME-CONFIRMATION QUESTIONS BEFORE CHECKING" rule to SCHEDULING block explicitly pointing to BEFORE BOOKING — READBACK as the single-confirmation moment.

**3B — Filler phrases too short.** "Let me check that for you" is ~1.5s while tool latency is 1-3s. When the filler ends and the tool hasn't returned, the caller says "Hello?" → server VAD fires → Gemini cancels the in-flight tool. Fix: TOOL NARRATION block now has Rule 3 ("AIM FOR ~3 SECONDS of speech") with ~3s example phrases for every tool (multiple variants per tool so the AI can naturally vary across a call).

**3C — Duplicate tenant fetch.** `check_availability` fetched the tenant config from Supabase every time, even though agent init already stashes the tenant in `deps`. Fix: reuse `deps["tenant"]`; fall back to a fresh fetch only if cached fields are missing. Saves 100-200ms per slot check.

**3D — Phase 60 directive bloat.** Every DIRECTIVE ended with a redundant "Do not repeat this message text on-air." trailer (STATE: prefix already signals machine-facing). Stripped 15 instances across all 5 tool files including split-across-literals variants. Smaller returns, less model context work.

### Scope explicitly excluded

- **Issue 1 (startup silence)** — user elected to skip for this hotfix.
- **Google Calendar timezone field** — a pre-existing bug from commit `e4a39a9` (2026-03-29), not a Phase 60 regression. User confirmed the specific complaint was the SMS, not the calendar invite email.
- **silence_duration_ms 1000→1500ms** — tradeoff (slower barge-in); apply only if Issue 3 fixes are insufficient in next UAT.

## Decisions Made

- **Two-layer defense on Issue 2** — Layer 1 (defensive parser) catches the bug even if Gemini misbehaves; Layer 2 (explicit STATE field names + directive) steers Gemini to emit correct format so Layer 1's fallback rarely fires. Together they eliminate the class of timezone-drift bugs.
- **Stripped the directive trailer via multiline regex** — the trailer was often split across adjacent string literals (e.g., `"Do not repeat this message"\n" text on-air."`), so line-based grep undercounted. Multiline regex caught 15 total instances.
- **Did not batch Issue 1 with Issue 2+3** — user explicitly scoped out the startup-silence fix for this hotfix.
- **Left Google Calendar timezone alone** — pre-existing bug, not in the user's reported complaint, and orthogonal to Phase 60.
