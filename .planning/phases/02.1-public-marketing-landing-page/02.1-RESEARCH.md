## Summary

This phase replaces `src/app/page.js` (currently a bare placeholder) with a full marketing landing page. The design system is already specified in the UI-SPEC document (02.1-UI-SPEC.md), copy is locked in the UI-SPEC Copywriting Contract, and the color palette is defined. Research focus is: how to wire Framer Motion scroll animations correctly in Next.js App Router, how to add Tailwind v4 CSS custom properties for the landing palette, how to structure the page as a Server Component with Client Component islands for animation, and what pitfalls to avoid.

**Key finding:** Framer Motion v12 is on npm (verified). It is NOT installed in this project — it must be added. Framer Motion requires `"use client"` components to run — the animated sections must be Client Components or wrapped in a Client boundary. The rest of the page can and should be a Server Component (page.js stays `async` server component, animation wrappers are separate client files).

**Key finding:** Tailwind v4 uses CSS custom properties defined in `globals.css` under `@theme inline` — NOT a `tailwind.config.js`. The landing palette (`--color-landing-*`) must be added to the `:root` block in `globals.css` and registered under `@theme inline` to be available as Tailwind classes.

**Primary recommendation:** Build the page as a Next.js Server Component (`page.js`) containing all static HTML sections; extract a single `AnimatedSection` client wrapper component that accepts children and applies `framer-motion` `whileInView` — this minimizes client bundle size while enabling all scroll animations.

## Architecture Patterns

### Recommended Project Structure

```
src/app/
├── page.js                          # Server Component — replace placeholder
└── components/landing/              # Landing-specific components (NOT in /ui/)
    ├── LandingNav.jsx               # "use client" — sticky scroll behavior
    ├── AnimatedSection.jsx          # "use client" — framer-motion whileInView wrapper
    ├── HeroSection.jsx              # Server Component — static content, uses AnimatedSection
    ├── HowItWorksSection.jsx        # Server Component — static 3-step sequence
    ├── FeaturesGrid.jsx             # Server Component — 4 Card components
    ├── SocialProofSection.jsx       # Server Component — 3 testimonial cards
    ├── FinalCTASection.jsx          # Server Component — big CTA block
    └── LandingFooter.jsx            # Server Component — logo + links
```

**Rationale for `src/app/components/landing/`:** The existing `/components/ui/` directory is for shadcn components. Landing-specific composite components belong in a separate folder. Using `src/app/components/` co-locates with the app while keeping a clear separation.

### Pattern 1: Server Component Page with Client Islands

**What:** The root `page.js` remains a Server Component (no `"use client"` directive). Each section that needs animation gets wrapped by `AnimatedSection` — a thin Client Component.

**When to use:** Always on this page. Keeps the landing page SSR-friendly, avoids shipping React hydration overhead for static copy.

```jsx
// src/app/page.js — Server Component (no directive needed)
import { LandingNav } from './components/landing/LandingNav';
import { HeroSection } from './components/landing/HeroSection';
// ... other section imports

export default function HomePage() {
  return (
    <>
      <LandingNav />
      <main>
        <HeroSection />
        <HowItWorksSection />
        <FeaturesGrid />
        <SocialProofSection />
        <FinalCTASection />
      </main>
      <LandingFooter />
    </>
  );
}
```

```jsx
// src/app/components/landing/AnimatedSection.jsx
'use client';
import { motion } from 'framer-motion';

export function AnimatedSection({ children, className, delay = 0 }) {
  return (
    <motion.div
      initial={{ opacity: 0, y: 24 }}
      whileInView={{ opacity: 1, y: 0 }}
      viewport={{ once: true }}
      transition={{ duration: 0.4, ease: 'easeOut', delay }}
      className={className}
    >
      {children}
    </motion.div>
  );
}
```

**Critical note on framer-motion v12:** The import path changed. Use `import { motion } from 'framer-motion'` — same as before. The `whileInView` + `viewport={{ once: true }}` pattern is unchanged and works in v12. `once: true` means the animation fires once and does not re-trigger on scroll-back — this is the correct behavior for a marketing page.

### Pattern 2: Sticky Nav with Scroll-Triggered Background

**What:** The nav starts transparent (on dark hero background, appears integrated), transitions to solid Midnight Slate with backdrop-blur after 20px of scroll.

**When to use:** LandingNav is a Client Component because it reads `window.scrollY`.

```jsx
// src/app/components/landing/LandingNav.jsx
'use client';
import { useState, useEffect } from 'react';
import Link from 'next/link';
import { Button } from '@/components/ui/button';

export function LandingNav() {
  const [scrolled, setScrolled] = useState(false);

  useEffect(() => {
    const handleScroll = () => setScrolled(window.scrollY > 20);
    window.addEventListener('scroll', handleScroll, { passive: true });
    return () => window.removeEventListener('scroll', handleScroll);
  }, []);

  return (
    <nav
      className={`fixed top-0 left-0 right-0 z-50 h-16 flex items-center justify-between px-6 transition-all duration-300 ${
        scrolled
          ? 'bg-[var(--color-landing-dark)] backdrop-blur-md shadow-md'
          : 'bg-transparent'
      }`}
    >
      {/* Logo left */}
      <span className="text-white font-semibold">HomeService AI</span>
      {/* CTA right */}
      <Button asChild size="sm" className="bg-[var(--color-landing-accent)] text-white hover:bg-[var(--color-landing-accent)]/90">
        <Link href="/onboarding">Start My 5-Minute Setup</Link>
      </Button>
    </nav>
  );
}
```

**Note on `{ passive: true }`:** Required per project skill rules (`client-passive-event-listeners`). Prevents janky scroll behavior on mobile.

### Pattern 3: Tailwind v4 Color Registration

**What:** The landing palette colors must be registered in `globals.css` in TWO places: the `:root` block (CSS variable values) AND the `@theme inline` block (makes them available as Tailwind utility classes).

**When to use:** Before any landing-specific Tailwind class like `bg-landing-surface` or `text-landing-accent` can work.

```css
/* src/app/globals.css — additions */

@theme inline {
  /* ... existing tokens ... */
  --color-landing-surface: var(--color-landing-surface);
  --color-landing-dark: var(--color-landing-dark);
  --color-landing-accent: var(--color-landing-accent);
  --color-landing-muted: var(--color-landing-muted);
  --color-landing-success: var(--color-landing-success);
}

:root {
  /* ... existing tokens ... */
  --color-landing-surface: #F5F5F4;
  --color-landing-dark: #0F172A;
  --color-landing-accent: #C2410C;
  --color-landing-muted: #475569;
  --color-landing-success: #166534;
}
```

**After this addition:** `bg-landing-surface`, `bg-landing-dark`, `bg-landing-accent`, `text-landing-muted`, `text-landing-success` become valid Tailwind v4 utility classes.

**Alternative (simpler, always works):** Use CSS custom property syntax directly in className — `className="bg-[var(--color-landing-accent)]"`. This works without `@theme inline` registration. Recommended for one-off usages; register in `@theme inline` for frequently used tokens.

**What:** The shadcn `Button` default variant uses `--primary` (near-black). Heritage Copper requires overriding via className or a new variant.

**When to use:** All 3 primary CTA placements + nav CTA.

### Anti-Patterns to Avoid

- **Adding `"use client"` to `page.js`:** Converts the entire page tree to client-side rendering, losing SSR benefits. Only the wrapper components that use browser APIs or Framer Motion need `"use client"`.
- **Using `<a href="/onboarding">` instead of `<Link>`:** Causes full page reload. Always use Next.js `Link` for internal routes.
- **Putting `AnimatedSection` as a Server Component:** `framer-motion` uses browser APIs (`IntersectionObserver`). Server rendering it will throw. It MUST have `"use client"`.
- **Using `viewport={{ once: false }}`:** Causes animations to re-trigger on scroll-back, which feels cheap on a marketing page. Use `once: true`.
- **Defining section components inside `page.js`:** Violates `rerender-no-inline-components` rule from project skills. Each section is its own file.
- **Using raw hex in Tailwind classes:** `className="bg-[#C2410C]"` works but bypasses the design token system. Use CSS custom property syntax `bg-[var(--color-landing-accent)]` or register in `@theme inline`.
- **Missing `min-h-[44px]` on touch targets:** WCAG 2.5.5 / project skill requirement — CTA buttons need minimum 44px height. The shadcn Button `lg` size (`h-10` = 40px) falls slightly short; override explicitly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Scroll-triggered animations | Custom IntersectionObserver + CSS class toggling | `framer-motion` `whileInView` | Framer handles RAF, reduced-motion, SSR hydration, stagger |
| Smooth-scroll to section | `window.scrollTo()` + offset calculation | CSS `scroll-behavior: smooth` + `href="#section-id"` | Native CSS smooth scroll works across all browsers; zero JS |
| Image optimization | `<img>` with manual width/height | `next/image` | Automatic WebP, responsive srcset, CLS prevention |
| Button with link behavior | `<button onClick={() => router.push()}>` | `<Button asChild><Link href="..."></Link></Button>` | asChild pattern gives proper semantics, prefetching, no full reload |
| Sticky nav scroll detection | `window.scrollY` polling in setInterval | `scroll` event listener with `passive: true` | Event-based is lower overhead; passive prevents paint blocking |

**Key insight:** On a static marketing page, the main complexity is CSS layout + animation wiring. Every "clever" custom solution adds bundle weight and bugs. Lean on what's already installed.

## Common Pitfalls

### Pitfall 1: Framer Motion v12 Breaking Change — Reduced Motion

**What goes wrong:** In framer-motion v12, the automatic `prefers-reduced-motion` respect was changed. If you set `transition.duration` without considering `useReducedMotion`, users with reduced motion preferences still see animations.

**Why it happens:** v12 moved some accessibility defaults — the library still respects `prefers-reduced-motion` at the CSS level when using `motion` components, but explicit `whileInView` configs can override it.

**How to avoid:** Use the `useReducedMotion` hook in `AnimatedSection`. If reduced motion is preferred, skip the animation variant:

```jsx
'use client';
import { motion, useReducedMotion } from 'framer-motion';

export function AnimatedSection({ children, className, delay = 0 }) {
  const shouldReduceMotion = useReducedMotion();

  return (
    <motion.div
      initial={shouldReduceMotion ? false : { opacity: 0, y: 24 }}
      whileInView={shouldReduceMotion ? {} : { opacity: 1, y: 0 }}
      viewport={{ once: true }}
      transition={{ duration: 0.4, ease: 'easeOut', delay }}
      className={className}
    >
      {children}
    </motion.div>
  );
}
```

**Warning signs:** Accessibility audit flags motion without `prefers-reduced-motion` support.

### Pitfall 2: `next-intl` Interference on `page.js`

**What goes wrong:** The current `page.js` uses `getTranslations` from `next-intl/server`. The landing page is primarily English marketing copy — no translations needed. But `layout.js` wraps children in `NextIntlClientProvider`, which is fine. The trap is accidentally leaving `getTranslations` in the new `page.js`, making the component async-dependent on locale resolution for content that doesn't need translation.

**Why it happens:** Copy-paste from existing page pattern.

**How to avoid:** The new `page.js` should NOT use `getTranslations`. All copy is hardcoded English strings inline (or passed as constants). The `NextIntlClientProvider` in `layout.js` is fine to leave in place — it doesn't affect pages that don't use `useTranslations`.

**Warning signs:** Page renders blank or throws during build if `getTranslations` is called without proper locale context.

---

### Pitfall 3: Tailwind v4 `@theme inline` Registration Order

**What goes wrong:** Adding CSS variables to `:root` without adding corresponding entries to `@theme inline` means the variables exist as CSS but are NOT usable as Tailwind utility classes (`bg-landing-accent` will be treated as unknown and purged).

**Why it happens:** Tailwind v4 changed from config-file approach to CSS-native approach — the `@theme inline` block is the equivalent of the old `extend.colors` in `tailwind.config.js`.

**How to avoid:** For every `--color-landing-*` variable, add TWO entries:
1. The value in `:root { --color-landing-surface: #F5F5F4; }`
2. The registration in `@theme inline { --color-landing-surface: var(--color-landing-surface); }`

**Alternative:** Use `bg-[var(--color-landing-surface)]` arbitrary value syntax — this always works without registration.

**Warning signs:** Tailwind class not applying at all; browser shows no generated CSS for that utility.

---

### Pitfall 4: Layout Shift from Missing Nav Height Offset

**What goes wrong:** The sticky nav is `h-16` (64px). The Hero section starts at `top: 0`. When the page loads, the hero content (especially the headline) is hidden behind the nav.

**Why it happens:** `position: fixed` elements are taken out of normal flow. The first section needs padding-top equal to the nav height.

**How to avoid:** Add `pt-16` (or `pt-[64px]`) to the Hero section's outermost container. This is noted in the UI-SPEC (96px desktop / 64px mobile for hero top padding).

**Warning signs:** CLS score is poor; headline text jumps on load.

---

### Pitfall 5: `page.js` Conflicts with `next-intl` Server Context

**What goes wrong:** The current `page.js` is an `async` function calling `getTranslations`. If the new landing page `page.js` is also `async` (for other reasons), it's fine. But if it's converted to a non-async Server Component that still has `getTranslations` in it, the build will fail due to missing async context.

**How to avoid:** The new `page.js` should be a simple synchronous Server Component (no `async`, no `getTranslations`). All content is static strings — no async fetching required.

---

### Pitfall 6: Smooth-Scroll Anchor Not Working with Fixed Nav

**What goes wrong:** `href="#features"` scroll brings the target section to the absolute top of viewport — hidden under the fixed 64px nav.

**Why it happens:** CSS `scroll-padding-top` is not set on the `<html>` element.

**How to avoid:** Add to `globals.css`:
```css
html {
  scroll-padding-top: 64px; /* matches nav height */
}
```
Or add it directly to `:root` in the `@layer base` block. This makes anchor link scroll land 64px below the actual target element.

## Code Examples

### AnimatedSection with Stagger (verified Framer Motion pattern)

```jsx
// Source: framer-motion whileInView API (framer.com/motion)
'use client';
import { motion, useReducedMotion } from 'framer-motion';

export function AnimatedSection({ children, className, delay = 0 }) {
  const prefersReducedMotion = useReducedMotion();

  return (
    <motion.div
      className={className}
      initial={prefersReducedMotion ? false : { opacity: 0, y: 24 }}
      whileInView={prefersReducedMotion ? {} : { opacity: 1, y: 0 }}
      viewport={{ once: true, margin: '-50px' }}
      transition={{ duration: 0.4, ease: 'easeOut', delay }}
    >
      {children}
    </motion.div>
  );
}
```

`viewport.margin: '-50px'` triggers the animation when the element is 50px into the viewport, preventing the flash-of-animated-content at the very bottom edge.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tailwind.config.js` extend.colors | `globals.css` `@theme inline` block | Tailwind v4 (2024) | No config file — CSS-native theming |
| `framer-motion` v10 `motion.div` JSX | Same API, `framer-motion` v12 (2025) | v11-v12 | Import path unchanged; some internal optimizations |
| shadcn CLI `--style flag` | `components.json` manual creation | shadcn v2 | Already handled in this project |
| `<a href>` for internal links | `<Link>` + `asChild` Button | Next.js App Router | Client-side nav, prefetching |
| CSS scroll-behavior polyfills | Native `scroll-behavior: smooth` | All modern browsers 2023+ | Zero JS needed for smooth scroll |

**Deprecated/outdated:**
- `framer-motion/dist/...` deep import paths: Use top-level `from 'framer-motion'` only in v12
- `AnimatePresence` is only needed for unmount animations — not required here since sections stay mounted
- `useAnimation` imperative API: Replaced by `whileInView` declarative pattern for scroll animations
