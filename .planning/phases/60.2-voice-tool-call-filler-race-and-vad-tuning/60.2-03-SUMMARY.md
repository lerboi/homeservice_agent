## Decisions Made

- **Filler-before-early-returns:** Every tool has early-return guards (`if not tenant_id`, `if not slot_start`, `if not owner_phone`). The filler is inserted BEFORE these — even error-path returns now emit a filler. Trade-off: caller briefly hears "Let me pull up the calendar..." even when the tool is going to fall back to "I'm having trouble checking availability right now." This is judged acceptable — the filler only runs on the happy path in practice, because tenant_id / slot_start / owner_phone are almost always populated. On genuine error paths, a filler + a brief error-recovery directive is still less jarring than silence + a brief error-recovery directive.
- **`deps` over `global`:** RESEARCH §R4 flagged that Railway workers handle concurrent calls in a single Python process. A module-level `global _filler_idx` would share rotation state across all concurrent callers, desyncing their per-tool phrase progression. `deps` is already per-session (constructed in `src/agent.py` for each entrypoint) and already the home of other per-call mutable state (`_tool_call_log`, `_booking_succeeded`, `_recovery_sms_fired`, `_last_booked_slot_key`). Adding `_filler_idx_<tool>` keys is zero-novelty.
- **Per-tool counter keys (not one shared key):** RESEARCH recommended independent rotation per tool. Implementation honored: a caller who hits `check_availability` 3× and `book_appointment` 1× sees the booking-filler at index 0, not index 3.
- **`except Exception`, not `except BaseException`:** Pitfall 3 — `asyncio.CancelledError` (subclass of BaseException, not Exception since Python 3.8+) must propagate on caller hangup so the tool task can unwind cleanly. Catching BaseException would swallow cancellation and risk worker hang.
- **`allow_interruptions=False` kept as documentation-only:** RESEARCH §R1 flagged that 1.5.1 silently downgrades this to `NOT_GIVEN` whenever the LLM is a RealtimeModel with server-side turn detection (our config). We pass it anyway so future readers understand the intent; the actual un-interruption lever is Plan 02's raised `silence_duration_ms=1500`.

## Deviations from Plan

**One deviation — test-helper extension (Rule 3: auto-fix blocking issue):**

The plan's `_minimal_deps()` was an empty dict `{}`, and `_invoke(tool_factory, mock_ctx, deps)` passed no kwargs. But:

1. **`supabase` access precedes tenant/slot guards.** All four tools access `supabase = deps["supabase"]` (bracket-access, raises `KeyError` on missing key) BEFORE their early-return guards. With `_minimal_deps() == {}`, every tool would crash on the supabase access and the filler-await assertion would be masked by `KeyError`. Fixed by adding `"supabase": MagicMock()` to `_minimal_deps()`.
2. **`transfer_call` needs `call_end_reason`.** After the filler and the `owner_phone` guard, `transfer_call` writes `deps["call_end_reason"][0] = "transferred"`. Tests that want the tool to run to completion (not just reach the filler) need this key. Added `"call_end_reason": [None]` to `_minimal_deps()`.
3. **`book_appointment` and `capture_lead` have REQUIRED positional args.** Python validates signatures BEFORE the function body runs — if required kwargs are missing, the tool never starts and the filler never plays. Added `_BOOK_APPOINTMENT_MIN_KWARGS` and `_CAPTURE_LEAD_MIN_KWARGS` constants (all empty strings) and passed them via `_invoke(..., **kwargs)` in the two affected tests. Empty strings trigger each tool's own early-return path (e.g., `if not slot_start or not slot_end: return "I need a bit more information..."`), so the tool reaches the filler await and exits cleanly without touching the MagicMock.

Commit `7f593dc` bundles the test-helper extension with Task 1 (where `supabase` access first became relevant). Plan 01's scaffold explicitly anticipated this: "_minimal_deps() returns an empty dict; Plan 03 will extend it per-tool as needed to reach the filler."

Tracked as `[Rule 3 - Blocking issue]` per GSD deviation rules. No user decision needed — this is plumbing that unblocks the agreed-upon acceptance tests.
