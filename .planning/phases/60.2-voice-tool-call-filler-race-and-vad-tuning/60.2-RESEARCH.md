1. **`allow_interruptions=False` is silently downgraded to `NOT_GIVEN` when using a RealtimeModel with server-side turn detection** (which Voco does). 2. **The module-level `global _filler_idx` counter in CONTEXT is buggy under concurrent calls** on the same Railway worker â€” one shared counter across all sessions. Move the counter into the per-session `deps` dict (already passed through the tool factory closure).

Additionally, the `_SegmentSynchronizerImpl.playback_finished` race is tracked as issue #4486 (OPEN since Jan 2026, no upstream fix). Document as accepted limitation.

## Architecture Patterns

### R1: `AgentSession.say()` API in livekit-agents 1.5.1

**Signature** `[VERIFIED: site-packages\livekit\agents\voice\agent_session.py:1019-1051]`:

```python
def say(
    self,
    text: str | AsyncIterable[str],
    *,
    audio: NotGivenOr[AsyncIterable[rtc.AudioFrame]] = NOT_GIVEN,
    allow_interruptions: NotGivenOr[bool] = NOT_GIVEN,
    add_to_chat_ctx: bool = True,
) -> SpeechHandle:
    ...
```

**Key properties (all verified from installed source):**

1. `say()` is **synchronous**, returns `SpeechHandle` directly (not a coroutine). `await session.say(...)` works because `SpeechHandle.__await__` is defined (see R5).
2. `allow_interruptions=False` IS a valid kwarg in 1.5.1. No API compatibility issue.
3. **CRITICAL GOTCHA** `[VERIFIED: agent_activity.py:905-914]`: when the underlying LLM is a `RealtimeModel` with `capabilities.turn_detection=True` (Voco's config; the google plugin sets `server_turn_detection=True` by default because we configure but don't disable `AutomaticActivityDetection`), `say()` **silently overrides** any `allow_interruptions=False` to `NOT_GIVEN`, logs a warning, and the resulting handle is interruptible:

   ```python
   # agent_activity.py:905-914 (installed 1.5.1)
   if (
       isinstance(self.llm, llm.RealtimeModel)
       and self.llm.capabilities.turn_detection
       and allow_interruptions is False
   ):
       logger.warning(
           "the RealtimeModel uses a server-side turn detection, "
           "allow_interruptions cannot be False when using VoiceAgent.say(), "
           "disable turn_detection in the RealtimeModel and use VAD on the "
           "AgentTask/VoiceAgent instead"
       )
       allow_interruptions = NOT_GIVEN  # silently dropped
   ```

   This means **Fix H's filler IS interruptible in practice** â€” the CONTEXT premise that "`allow_interruptions=False` guarantees the filler plays to completion" is incorrect against 1.5.1 + our Gemini config. `[VERIFIED: installed source]`

4. **Why Fix H still helps anyway:** Fix H's real win is **determinism**. Today the filler is prose in the system prompt, and Gemini generates both the filler text AND the `function_call` in the same turn â€” so they race at the text-generation layer. With Fix H, the filler TEXT is a fixed string scheduled via `activity._tts_task(...)` BEFORE the tool body runs. Even if the filler is later interrupted, it cannot be TRUNCATED mid-first-syllable by Gemini's own self-interrupt (which is what caused `"Alright, let me go [cut]"` in the UAT transcript). The race that Fix H eliminates is **not** "interrupt by caller" but **"interrupt by Gemini's own new generation event when the tool response arrives."** Fix G (1500ms silence_duration) is what reduces caller-triggered interruptions.

5. `RunContext.session` IS an `AgentSession` `[VERIFIED: events.py:53-55]`:

   ```python
   @property
   def session(self) -> AgentSession[Userdata_T]:
       return self._session
   ```

6. **Exceptions `say()` can raise synchronously:**
   - `RuntimeError("AgentSession isn't running")` â€” only if `session._activity is None`, effectively unreachable inside a tool (tool ran â†’ session is running)
   - `RuntimeError("AgentSession is closing, cannot use say()")` â€” only if the session is tearing down mid-tool (rare; happens on caller hangup during tool execution)
   - TTS-model-missing `RuntimeError("trying to generate speech from text without a TTS model")` `[VERIFIED: agent_activity.py:897-903]` â€” NOT a risk for Voco since Gemini Live handles audio output directly, and `self._session.output.audio_enabled` check short-circuits when using realtime model
   - **Async exceptions (during `await`):** cancellations (if the whole tool task is cancelled), anything raised inside the TTS pipeline
   - **Recommendation:** the CONTEXT's `try/except Exception as e: logger.warning(...)` pattern is correct and sufficient. Don't narrow the except clause â€” multiple RuntimeError sources and async cancellation all benefit from the same "log and continue" path.

### R5: Does `await session.say()` block until audio playout?

**Yes, via `SpeechHandle.__await__` â†’ `wait_for_playout()` â†’ `_done_fut`.** `[VERIFIED: speech_handle.py:156-184]`

```python
# speech_handle.py:179-184
def __await__(self) -> Generator[None, None, SpeechHandle]:
    async def _await_impl() -> SpeechHandle:
        await self.wait_for_playout()
        return self
    return _await_impl().__await__()
```

`wait_for_playout()` awaits `self._done_fut`, which is marked done when the TTS playout completes OR the handle is interrupted (via `_cancel()`).

**Circular-wait check** `[VERIFIED: speech_handle.py:156-178]`: `wait_for_playout` raises a `RuntimeError` if called from inside the function tool that owns the handle being waited on. **For Fix H this is safe** because `session.say()` creates a FRESH `SpeechHandle` (via `SpeechHandle.create()` in `agent_activity.py:916`) that is NOT the tool's parent handle. The check compares `info.speech_handle == self` (line 169) â€” fresh handle â‰  parent handle â†’ no error.

**Scheduling behavior** `[VERIFIED: agent_activity.py:1142-1181]`: all speeches go through a serial priority heap processed by `_scheduling_task`. `say()` enqueues at `SPEECH_PRIORITY_NORMAL` (same as AI replies). If the AI is mid-reply when the tool fires `session.say()`, the filler will not play until the parent reply's speech-handle is done. **Practical implication:** the "parent reply" in Voco's case is whatever audio was playing when Gemini emitted the `function_call`. On Gemini Live, `_handle_tool_calls` marks the current generation done (`_mark_current_generation_done()` at `realtime_api.py:1293`), which unblocks the scheduling queue so the filler can run. Timing is tight but correct.

**Tool-duration impact:** `await session.say("...")` blocks the tool coroutine for the duration of the filler audio (~2-3 seconds for the CONTEXT's phrases at Gemini's ~150 wpm rate). This is FINE â€” as CONTEXT notes, "we're covering latency that was ALREADY silence."

### R4: Rotating filler state â€” use `deps`, not `global`

**CONTEXT's example uses `global _filler_idx`. Do not do this.** The agent's Railway worker handles MULTIPLE CONCURRENT CALLS in a single process. A module-level `global` counter is shared across all sessions â†’ two calls hitting `check_availability` simultaneously will both increment the same counter, effectively rotating twice. Worse, the counter NEVER resets, so across thousands of calls it just drifts to a large integer (not a correctness issue thanks to modulo, but semantically muddled).

**Recommended pattern** â€” use the existing `deps` dict passed through the tool factory closure (already per-session, already mutable, already the place where other per-call state like `_tool_call_log`, `_booking_succeeded` lives):

```python
# Top of each tool file (module-level)
_FILLER_PHRASES = [
    "Let me pull up the calendar for you real quick â€” one moment.",
    "Give me just a second to look at what we have open.",
    "Let me take a look at the schedule for you â€” one sec.",
]

def create_check_availability_tool(deps: dict):
    # Initialize per-session counter once (set inside the factory, runs per call)
    deps.setdefault("_filler_idx_check_availability", 0)

    @function_tool(name="check_availability", description="...")
    async def check_availability(context: RunContext, ...) -> str:
        # Phase 60.2 Fix H: deterministic filler audio.
        idx = deps["_filler_idx_check_availability"]
        phrase = _FILLER_PHRASES[idx % len(_FILLER_PHRASES)]
        deps["_filler_idx_check_availability"] = idx + 1
        try:
            await context.session.say(phrase, allow_interruptions=False)
        except Exception as e:
            logger.warning("[check_availability] filler say() failed: %s", e)

        # ... rest of tool body unchanged ...
```

**Why this works:**
- `create_check_availability_tool(deps)` is called once per entrypoint invocation (i.e., once per call) â€” see `src/agent.py:203` and `src/tools/__init__.py`. `deps.setdefault(...)` seeds the counter on first tool creation.
- Each tool gets its own key (`_filler_idx_check_availability`, `_filler_idx_book_appointment`, etc.) so phrases rotate independently per tool, even when the same caller triggers multiple tools in one call.
- No thread-safety concerns: Python's GIL + asyncio single-threaded-per-task means read-modify-write on a dict key in a single coroutine step is atomic.

### Anti-Patterns to Avoid

- **Module-level `global` filler counter** â€” as above, wrong under concurrent calls per worker.
- **Bare `except:`** in the filler try/except â€” catching `BaseException` risks swallowing `asyncio.CancelledError` from tool-task cancellation. CONTEXT's `except Exception as e:` is correct.
- **Awaiting `handle.wait_for_playout()` from inside the tool that owns `handle`** â€” raises circular-wait RuntimeError. (Not a risk with Fix H because we create a new handle via `session.say()`, but worth knowing if future work tries to wait on the PARENT handle.)
- **Passing `allow_interruptions=False` and assuming it takes effect on a RealtimeModel session** â€” silently overridden, warning logged. Don't rely on it for guaranteed-uninterruptible playback.

## Common Pitfalls

### Pitfall 1: `allow_interruptions=False` silently dropped

**What goes wrong:** Developer writes `session.say(text, allow_interruptions=False)`, expects guaranteed uninterruptible playback. It isn't.
**Why it happens:** `agent_activity.py:905-914` hardcodes an override when the LLM is a `RealtimeModel` with server-side turn detection.
**How to avoid:** Understand that Fix H's value is DETERMINISM of filler TEXT (vs. LLM prose generation), not UN-INTERRUPTIBILITY of filler AUDIO. Fix G (VAD tuning) is what handles the interruption side.
**Warning signs:** The log line `"the RealtimeModel uses a server-side turn detection, allow_interruptions cannot be False..."` will appear on every tool invocation once Fix H ships. This is noisy but benign â€” if desired, suppress by leaving `allow_interruptions` unspecified (defaults to `True`, effectively the same outcome).

### Pitfall 2: `global _filler_idx` across concurrent sessions

**What goes wrong:** Two calls on the same Railway worker hit `check_availability` within the same second; counter increments twice instead of once-per-call; rotation is out-of-sync per call.
**Why it happens:** Module globals are process-scoped, not session-scoped.
**How to avoid:** Use `deps` (per-session, already plumbed through tool factories).
**Warning signs:** Hard to detect from logs â€” manifests as "rotation feels random" under production load. Not worth special monitoring; just use `deps` from day one.

### Pitfall 3: Tool-task cancellation during `await session.say(...)`

**What goes wrong:** Caller hangs up while filler is playing. The session starts tearing down; tool task is cancelled mid-`await session.say()`. `asyncio.CancelledError` propagates out of the `say()` await.
**Why it happens:** Normal asyncio cancellation semantics.
**How to avoid:** The CONTEXT's `try/except Exception as e:` handles this correctly â€” `CancelledError` is a subclass of `BaseException` (Python 3.8+), NOT `Exception`, so the except does NOT swallow it. Cancellation propagates as intended. **Don't broaden the except to `BaseException`** â€” that would suppress cancellation and hang the worker.
**Warning signs:** Any "filler say() failed: CancelledError" log is concerning â€” should be absent.

### Pitfall 4: `check_customer_account.py` reminded as NOT in scope

CONTEXT says only 4 tool files get Fix H. `check_customer_account.py` is a pre-session data tool (context merge from Jobber/Xero) that runs before the AI speaks. It should NOT get a filler â€” doing so would interfere with the greeting path. Research confirms this is correct; the tool signature and post-return flow differ from the 4 scoped tools.

## Fix I Triage: Post-call 8s timeout

**Location** `[VERIFIED: C:\Users\leheh\.Projects\livekit-agent\src\agent.py:293-330]`:


**When introduced:** Commit `fe96e805` on **2026-04-14 05:29:09 +0800** (git blame verified). NOT new. The timeout has been in place for 5+ days before the 2026-04-19 UAT observation.

**What the 8s wraps** (`run_post_call_pipeline()` at `src/post_call.py:24-429`) â€” strictly serialized (not gathered):
1. Build transcript text/structured
2. Supabase `calls` UPDATE (status, end_timestamp, transcript, disconnection_reason)
3. Supabase `calls` SELECT (fetch updated row)
4. Booking reconciliation: `calls` UPDATE + `appointments` UPDATE (only if `booking_succeeded`)
5. Test-call auto-cancel: `appointments` UPDATE + `leads` UPDATE (only if `is_test_call`)
6. Usage tracking: `increment_calls_used` RPC + optional Stripe usage record (overage path)
7. Language detection (local regex only, no I/O)
8. **Triage classification: OpenAI LLM call via `classify_call()`** â€” the expensive one
9. Suggested slots calculation (DB-heavy if `booking_outcome in (null, "not_attempted")`)
10. `calls` UPDATE with triage/language/slots
11. `calls` UPDATE booking_outcome NULL-fallback
12. Hallucination detection (2 `calls` reads/updates if patterns match)
13. Lead create-or-merge (2-5 DB operations inside `create_or_merge_lead`)
14. Tenant fetch (1 `tenants` SELECT)
15. Final `calls` SELECT for booking_outcome
16. Owner notification: SMS (Twilio) + email (Resend) via `asyncio.gather` (the only parallelism)

Even in the happy path, that's **~14 serial Supabase round trips + 1 OpenAI call + Stripe + Twilio + Resend**. At 300-500ms per hop (Railway â†’ Supabase â†’ RLS is 150-250ms; OpenAI classify is 1-3s; Twilio SMS is 400-800ms), total easily exceeds 8s on calls with long transcripts.

**Is it new?** The TIMEOUT didn't change. The pipeline got SLOWER between Apr 14 and Apr 19:
- `c307952` (2026-04-16) "P56-06: merge helper + concurrent Jobber+Xero context fetch" â€” affects pre-session, not post-call; IRRELEVANT to this timeout
- `b9688c8` "updated" â€” unknown scope
- Nothing in post-call itself got slower per git log

**Actual cause of the 8s breach in the UAT:** The transcript on that specific call was ~193s long with repeated agent self-corrections (from the 2 server-cancelled tool calls â†’ agent retried phrases â†’ double-length transcript). Longer transcript â†’ larger OpenAI classify prompt â†’ slower triage â†’ pipeline busts 8s.

**Is it only on cancellation-heavy calls or every call?** Evidence from one UAT call is insufficient for certainty, but the log sequence strongly implies correlation: `"server cancelled tool calls"` twice + `"book_appointment: idempotent re-invocation"` + `"_SegmentSynchronizerImpl.playback_finished"` + TIMEOUT. A clean call with a short transcript and single book_appointment likely completes in <5s.

**Verdict: DEFER.** The 8s timeout is not a regression; it's a pre-existing safety belt that's now being hit because the cancellations inflate the transcript. Fixing G+H should shrink transcripts back to normal, pulling triage-LLM latency back under the threshold. Re-measure after 60.2 deploys. If timeouts persist on clean calls, open a follow-up phase to (a) parallelize post-call stages that don't depend on triage output (steps 2-3, 4, 5, 6 can all run concurrently with 8 via `asyncio.gather`), or (b) raise the `shutdown_process_timeout` + this timeout in lockstep.
