# Phase 60.2 Context — Voice Tool-Call Filler Race + Server-VAD Tuning

This file exists so this phase can be picked up in a fresh chat with zero prior context. Read the whole thing before editing anything.

## TL;DR

Phase 60 landed voice-agent prompt polish. Phase 60.1 (hotfix) fixed a caller-SMS wrong-time regression and reduced tool-call voice glitches from 4 `server cancelled tool calls` per call → 2 per call by trimming directives, lengthening prompt fillers, killing over-confirmation, and caching the tenant fetch.

**Remaining problem:** The 2 cancellations per call produce audible glitches during tool-call handling — the AI cuts itself off mid-sentence ("Alright, let me go [cut]" → blank → "Perfect, that's all booked in [cut]"). Root cause is a race inside the LiveKit+Gemini integration that **prompt changes alone cannot win**. This phase applies two deterministic fixes that bypass the race.

**Two fixes to implement:**
1. **Fix G** — Raise Gemini server-VAD `silence_duration_ms` from `1000` to `1500` in `src/agent.py`. Catches server-VAD cancellations from caller "Hello?"/breath interjections during tool-execution silence windows. Small tradeoff: caller barge-in takes ~0.5s longer to register.
2. **Fix H** — Replace prompt-based tool-call filler with programmatic `context.session.say(filler, allow_interruptions=False)` as the first line of each tool function. Guarantees filler audio plays to completion before the tool body runs, eliminating the filler-vs-tool-response race.

Plus two investigative tasks with less obvious payoff (decide during planning whether to scope into this phase or defer):
3. **Fix I (investigate)** — Post-call pipeline started hitting an 8s timeout in the latest UAT log (`Post-call pipeline TIMEOUT after 8s — partial writes possible`). This is NEW — first observed 2026-04-19 15:35:36Z. Triage whether it's a cascade from the cancellations or an independent regression.
4. **Fix J (research)** — SKILL.md §1 notes that `livekit-agents==1.5.1` and the `livekit-plugins-google` git pin at commit `43d3734` are hard-pinned because newer plugin versions constructed `RealtimeCapabilities` with the wrong field count. Investigate whether a newer plugin commit (or livekit-agents 1.6+ line) exists that preserves `gemini-3.1-flash-live-preview` support AND fixes the `_SegmentSynchronizerImpl.playback_finished called before text/audio input is done` race. If yes, upgrade. If no, document as accepted.

## Why prompt fixes alone cannot fix this

Phase 60.1 already:
- Stripped "Do not repeat this message text on-air." trailer (15 instances)
- Rewrote filler examples from ~1.5s to ~3s
- Killed "Just to confirm, X?" pre-tool over-confirmation
- Cached the tenant dict in `deps` (saves 100-200ms per `check_availability`)

These are all helpful. What they **cannot** win:

```
  t=0     Gemini emits filler text + function_call in parallel
  t=0.2s  Filler audio starts playing to caller
  t=0.3s  function_call reaches tool runtime
  t=0.5s  Tool returns (fast! — cache hit or simple RPC)
  t=0.5s  function_response goes back to Gemini
  t=0.6s  Gemini begins generating post-tool response text
  t=0.6s  Gemini server-side: "I have a new turn to speak" → interrupts current audio
  t=0.6s  Filler audio TRUNCATED mid-sentence
  t=1.0s  Caller hears "Alright, let me go [silence]"
```

The faster the tool returns, the worse this race. And Phase 60.1 made `check_availability` faster by caching the tenant — the silver lining is also the problem.

**No amount of longer-filler prompting can win this race, because the race is between audio playback and tool-response-induced new-generation — both driven by Gemini server events the model cannot control via text-level instructions.**

## Fix G — Raise server-VAD `silence_duration_ms` 1000 → 1500ms

**File:** `src/agent.py`
**Lines (current state after Phase 60):** ~231-238

```python
# Dampen Gemini's server-side VAD so it stops cancelling in-flight tool
# calls on breaths or minor overlap (backlog 999.2). LOW sensitivity +
# larger prefix/silence padding tracks upstream guidance for
# livekit/agents#4441 ("Spurious Server VAD events cause unavoidable
# tool cancellation"). Barge-in still works — the thresholds just
# require deliberate caller speech instead of firing on breath/noise.
realtime_input_config = genai_types.RealtimeInputConfig(
    automatic_activity_detection=genai_types.AutomaticActivityDetection(
        start_of_speech_sensitivity=genai_types.StartSensitivity.START_SENSITIVITY_LOW,
        end_of_speech_sensitivity=genai_types.EndSensitivity.END_SENSITIVITY_LOW,
        prefix_padding_ms=400,
        silence_duration_ms=1000,   # ← change to 1500
    ),
)
```

**Change:** `silence_duration_ms=1000` → `silence_duration_ms=1500`.

**Why:** Server VAD interprets any caller-side audio longer than `silence_duration_ms` after the AI goes silent as "end of caller turn → send to model → potentially cancel in-flight function calls." Raising from 1000ms → 1500ms means brief "Hello?" or "Done" fillers from the caller during tool-execution silence no longer trigger cancellation.

**Tradeoff:** Caller barge-in (caller intentionally interrupts the AI) takes ~500ms longer to register. Acceptable — the UAT call shows the AI is already deferential when interrupted; adding 500ms of "let the AI finish" is a net win for the common case.

**Update the comment** above the config to note the 1500ms change and reference this phase.

**Verification:** Re-run UAT Persona 1. The `server cancelled tool calls` warnings should drop from 2/call → 0-1/call in normal conversation. Barge-in still works when the caller speaks deliberately for >1.5s.

## Fix H — Programmatic `session.say()` filler inside each tool

**Files:**
- `src/tools/check_availability.py`
- `src/tools/book_appointment.py`
- `src/tools/capture_lead.py`
- `src/tools/transfer_call.py`

(NOT `check_caller_history.py` — it's silent-context, no filler needed.
NOT `end_call.py` — its return is a space character, no filler.)

**Pattern to implement (example for `check_availability`):**

```python
async def check_availability(
    context: RunContext,
    date: str = "",
    time: str = "",
    urgency: str = "routine",
) -> str:
    # Phase 60.2 Fix H: programmatic filler audio.
    # This REPLACES reliance on the prompt to emit a filler before tool
    # invocation. allow_interruptions=False means Gemini's own self-interrupt
    # (triggered when the tool response arrives mid-audio-playback) cannot
    # truncate this audio — it plays to completion. Eliminates the
    # "Alright, let me go [cut]" race observed in the 2026-04-19 UAT call.
    try:
        await context.session.say(
            "Let me pull up the calendar for you real quick — one moment.",
            allow_interruptions=False,
        )
    except Exception as e:
        # Never block tool execution on filler audio failure.
        logger.warning("[check_availability] filler say() failed: %s", e)

    # ... rest of tool body unchanged ...
```

**Rotating phrases.** To avoid the model sounding robotic when the same tool is invoked 2-3 times in a conversation, rotate through a small list. Example pattern:

```python
# Module-level constant near the top of each tool file
_FILLER_PHRASES = [
    "Let me pull up the calendar for you real quick — one moment.",
    "Give me just a second to look at what we have open.",
    "Let me take a look at the schedule for you — one sec.",
]
_filler_idx = 0

async def check_availability(...):
    global _filler_idx
    phrase = _FILLER_PHRASES[_filler_idx % len(_FILLER_PHRASES)]
    _filler_idx += 1
    try:
        await context.session.say(phrase, allow_interruptions=False)
    except Exception as e:
        logger.warning("[check_availability] filler say() failed: %s", e)
    # ...
```

**Per-tool filler phrases (use these verbatim; already validated in Phase 60.1 TOOL NARRATION block):**

| Tool | Filler phrases (rotate) |
|------|-------------------------|
| `check_availability` | "Let me pull up the calendar for you real quick — one moment." / "Give me just a second to look at what we have open that day." / "Let me take a look at the schedule for you — one sec." |
| `book_appointment` | "Alright, let me go ahead and lock that in for you now." / "Let me get that booked in for you — give me just a second." / "Perfect, booking that slot now — one moment." |
| `capture_lead` | "Let me make a note of your details so the team can follow up." / "Let me get all that saved down for you — one second." |
| `transfer_call` | "Let me get you through to someone on the team — one moment." / "Connecting you over now, just a second." |

**Once Fix H is in place, update the prompt to reflect reality:**

- `src/prompt.py` `_build_tool_narration_section()` (~line 112): add a note that filler audio is now handled automatically by the tool runtime, and the prompt's role is to ensure the AI doesn't generate a DIFFERENT filler that would overlap. New instruction: "Do NOT speak a filler phrase yourself — the tool runtime plays one automatically. Just invoke the tool."
- Existing filler examples can stay as fallback prose but their criticality drops.

**Verification:**
1. The "Alright, let me go [cut]" transcript pattern should disappear.
2. The filler audio should play to completion every time, even when the tool returns in <100ms.
3. `book_appointment: idempotent re-invocation` log messages should drop to zero (no more cancel-retry loops).

**Risk:** `context.session.say()` blocks the tool function until audio finishes playing. For a 3-second filler, that adds 3 seconds of latency to the tool's wall-clock execution time. But since we're covering latency that was ALREADY silence, this isn't user-perceived latency — it's just the tool runtime using the silence window productively.

**Verify `session.say()` signature against the installed `livekit-agents==1.5.1`** before writing code. The API is:

```python
# from livekit.agents.voice import AgentSession
# context.session is an AgentSession; .say() returns SpeechHandle
handle = await context.session.say(
    text="...",
    allow_interruptions=True,  # default — we want False
)
# handle is a SpeechHandle; await handle.wait_for_playout() if you need to know when audio finished.
```

If `allow_interruptions=False` is not available in 1.5.1 (check the installed version's source), the alternative is to call `session.say()` then `await handle.wait_for_playout()` to block until audio finishes. Functionally equivalent.

## Fix J (research) — Can we upgrade the google plugin pin?

**Context from SKILL.md §1:**

> **Pin fix:** `livekit-agents`, `livekit-plugins-silero`, `livekit-plugins-turn-detector` locked to `==1.5.1` in `pyproject.toml`. PyPI `livekit-agents 1.5.2` shipped Apr 8 2026 with PR #5211, which added a required 7th field `per_response_tool_choice` to `llm.RealtimeCapabilities`. The git-pinned google plugin at commit `43d3734` still constructs `RealtimeCapabilities` with 6 fields, so any Railway rebuild after Apr 8 produced a `TypeError` at `RealtimeModel.__init__` on every inbound call. The plugin pin must stay because commit `43d3734` is the only google plugin version supporting `generate_reply()` with `gemini-3.1-flash-live-preview` (via the `A2A_ONLY_MODELS` branch).

2. Check whether `livekit-agents 1.6+` has shipped AND whether the google plugin has been updated to match the newer `RealtimeCapabilities` shape AND whether that newer version still supports `gemini-3.1-flash-live-preview`.
3. If a compatible newer commit exists, update both pins in `pyproject.toml` and test in a Railway preview deploy.
4. If no compatible version exists, accept the `_SegmentSynchronizerImpl` race as a known limitation and add a note to SKILL.md.
