## Architecture (Phase 64+ — pipeline)

```
Twilio SIP → LiveKit room → AgentSession(
    stt=google.STT(chirp_3, languages=<bcp47>, detect_language=False),
    llm=google.LLM(gemini-3.1-flash, thinking_config=ThinkingConfig(low)),
    tts=GeminiTTS(Zephyr, gemini-2.5-flash-preview-tts),
    vad=silero.VAD.load(min_silence_duration=2.5),
) → post-call pipeline (transcript + triage + booking + notifications)
```

### Why pipeline (not Realtime audio-to-audio)
- Realtime `gemini-3.1-flash-live-preview` capability-gates `session.say()`,
  `session.generate_reply()`, and `update_chat_ctx()` closed (mutable_chat_context=False
  for 3.1 models). Pipeline has no capability gates.
- Realtime path has an unfixed mid-word truncation race
  (`_SegmentSynchronizerImpl`, livekit/agents#4486/#5096). Pipeline does not use
  that synchronizer.
- Realtime path has unfixed `server cancelled tool calls` race
  (livekit/agents#4441) where caller VAD fires during tool execution. Pipeline
  tool lifecycle is framework-local (not Gemini-server-cancellable).
- Tradeoff: +~500ms per-turn latency (audio-to-audio ~300ms → pipeline ~800-1200ms)
  in exchange for definitively reliable behavior.

### Component map
- STT: google.STT — Chirp 3 with BCP-47 language pin, detect_language=False for
  tenant-locale enforcement (Phase 60.4 Stream B carry-over via kwarg rename
  `language=` → `languages=`). Pitfall 1: plural kwarg required, singular silently
  discarded.
- LLM: google.LLM — gemini-3.1-flash (text mode, same reasoning as 3.1 Flash Live),
  thinking_config(level=low, include_thoughts=False). Accepts 7 @function_tool
  tools identically to Realtime — return string, insert as FunctionCallOutput, LLM
  called again.
- TTS: GeminiTTS (plugin existed pre-Phase-64 as greeting-only; promoted to
  session-level in Phase 64). Zephyr default voice.
- VAD: silero.VAD.load(min_silence_duration=2.5) — local, ~30ms, free. Ports the
  Phase 63.1-11 2500ms silence threshold from realtime_input_config.

### Tool lifecycle
Pipeline tools execute via `AgentActivity.perform_tool_executions()`:
1. LLM emits FunctionToolCall in LLMStream
2. perform_tool_executions runs the tool coroutine
3. Result inserted into chat_ctx as FunctionCallOutput
4. LLM called again with tool result in context

No server-side cancellation pathway (unlike Realtime's Gemini-server-VAD-initiated
tool cancellation). Tools run atomically once invoked.

### Greeting dispatch
```python
greeting_handle = session.say(greeting_text)  # fire-and-forget
```
Pipeline routes `say()` through `_tts_task` when `tts=` is attached. No capability
gate. The greeting is the assistant's first turn in chat history — `_build_greeting_section`
directs the LLM to respond to caller input on its next turn (not re-greet).
