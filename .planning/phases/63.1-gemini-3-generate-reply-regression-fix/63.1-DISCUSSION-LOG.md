## Approach selection

| Option | Description | Selected |
|--------|-------------|----------|
| #1 Pure system-prompt greeting | Add "START BY GREETING THE CALLER" directive inside `build_system_prompt`; rely on Gemini's `instructions=system_prompt` at RealtimeModel construction | |
| #2 Synthetic user turn | Push a fake user turn via 1.5.6 API after `session.start()` to trigger the model's greeting | |
| #3 Construction-time intake injection only | Move DB intake-question fetch before `session.start()`; inject into `system_prompt`. Accept silent-connect for greeting. | |
| #4 Hybrid: #3 + #1 | Intake via #3; greeting via pure prompt | |
| #4 Hybrid: #3 + #2 | Intake via #3; greeting via synthetic user turn | ✓ |

**User's choice:** Hybrid #3 + #2.
**Notes:** Decision driven by caller-UX framing — silent connect reads as "broken call" to a real customer, especially on emergency calls (plumbing 2am). Synthetic user turn is the most reliable greeting trigger because Gemini Live's server VAD is designed to respond to user turns. Research spike (Plan 01 Task 1) verifies exact 1.5.6 API surface; #1 is the fallback if no push-transcription API exists.

## Connect-latency budget

| Option | Description | Selected |
|--------|-------------|----------|
| A. Unbounded | Accept ~300-500ms DB-query latency before greeting | ✓ |
| B. Keep parallel | Find a way to keep intake fetch parallel with session start | |
| C. Capped at 200ms | Skip intake injection if DB is slow; log | |

**User's choice:** A — unbounded.
**Notes:** Latency is invisible to the caller (sits between SIP connect and agent's first word). Real receptionists take 1-2s to say hello; an extra 500ms is imperceptible and actually feels more natural. Correctness (intake questions reach the model) beats micro-latency.

## Regression prevention

| Option | Description | Selected |
|--------|-------------|----------|
| A. Grep-based pytest | Static test scans `src/` for `session.generate_reply(` and fails if found | ✓ |
| B. Runtime assertion | Raise at runtime if `RealtimeModel` + `generate_reply` combo detected | |
| C. Documentation only | Memory note + code comment | |

**User's choice:** A — grep-based pytest.
**Notes:** Zero-runtime-overhead, catches the exact anti-pattern, plays cleanly with existing pytest workflow. Grep must be narrow enough to permit `generate_reply` in tests and comments — only flag active invocations in `src/`.
