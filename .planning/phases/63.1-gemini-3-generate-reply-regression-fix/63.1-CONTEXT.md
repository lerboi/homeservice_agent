<decisions>
## Implementation Decisions

### Approach (Q1)
- **D-01:** **Hybrid approach.** Two independent fixes applied together:
  - **Intake injection (`src/agent.py:712`)** → **Approach #3** (construction-time injection). Reorder so the DB query that fetches `intake_questions` completes BEFORE `session.start()`, then inject the questions into `system_prompt` passed to `RealtimeModel(instructions=system_prompt)`. Eliminates the `generate_reply` call entirely for intake.
  - **Opening greeting (`src/agent.py:755`)** → **Approach #2** (synthetic user turn). After `session.start()` completes, push a synthetic user turn via the 1.5.6-sanctioned API (likely `session.input.transcription` or equivalent — research spike to verify exact surface). This triggers Gemini's server-VAD to fire the greeting from `instructions=system_prompt` within ~500ms of SIP connect.
  - **Fallback:** If the synthetic-user-turn API does not exist on 1.5.6, fall back to Approach #1 (pure system-prompt directive) for greeting — accept the silent-connect risk in exchange for a shipping path. Research spike decides.

### Latency (Q2)
- **D-02:** **Connect-latency budget = unbounded (within DB round-trip).** Moving DB queries before `session.start()` adds ~300-500ms before the caller hears anything. This is explicitly acceptable — correctness wins over micro-latency. The extra delay is imperceptible vs. natural human phone-pickup timing (humans take 1-2s to say hello) and actually makes the agent feel more natural, not less.

### Testing bar (Q3)
- **D-03:** **Local pytest + feature branch + Railway preview + one live UAT call.** Same rigor as Phase 63 — we just shipped Phase 63 on this bar and caught the `generate_reply` regression on the first real call, so the bar is working. Do NOT lower it. Merge to `livekit-agent/main` only after: all tests green, Railway preview deploy reports SUCCESS + "registered worker", one UAT call hears the agent greet FIRST without caller prompting, and (if the tenant has intake questions configured) at least one intake question surfaces naturally during the call.

### Regression prevention (Q4)
- **D-04:** **Grep-based pytest guard.** Add a test under `livekit-agent/tests/` that scans `src/agent.py` (and `src/` broadly) for `session.generate_reply(` calls and fails if any are found. Rationale: a runtime assertion risks breaking a different code path unpredictably; a static test is zero-runtime-overhead, catches the exact pattern we want to prevent, and plays cleanly with the existing pytest-on-livekit-agent workflow. The grep should be narrow enough to permit `generate_reply` in tests and comments (only flag active invocations).

<specifics>
## Specific Ideas

### Caller UX anchor (why hybrid, not pure-prompt)
From the caller's perspective, a 3-5+ second silent pause after SIP connect reads as "broken call." Phase 63's UAT transcript (call `call-_+6587528516_XwbNh8W3js2h`) shows the user had to say "Hello" first to wake the agent — a real customer, especially on an emergency plumbing call, will hang up before saying "hello?" into silence. Approach #2 (synthetic user turn) solves this definitively because Gemini Live's server VAD is designed to respond to user turns — injecting a synthetic one is how the SDK ecosystem bootstraps "agent speaks first" flows.

### Latency framing
The 300-500ms DB-query latency added by Approach #3 is invisible — it sits between SIP connect and the agent's first word. Real receptionists don't answer in 100ms; an extra half-second makes the agent sound *more* human.

### Naming
`phase-63.1-generate-reply-fix` for the branch. `fix(63.1):` commit prefix in livekit-agent; `docs(63.1-XX):` prefix in homeservice_agent for planning artifacts. `--no-verify` on both repos per established convention.
</specifics>
