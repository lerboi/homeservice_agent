# Phase 63.1 — Gemini 3 `generate_reply` Regression Fix (seed)

This is a **seed** for `/gsd:discuss-phase 63.1`. It is the minimal context a discussion agent needs to ask the right follow-up questions, not the final decision log.

## Problem (observed)

Phase 63 shipped the livekit-agents + livekit-plugins-google upgrade to `==1.5.6` (merge commit `9ce12d6` on `livekit-agent/main`, merged 2026-04-23). The first post-merge UAT call surfaced an immediate production regression: **`session.generate_reply()` is silently dropped by livekit-plugins-google 1.5.6 when the active `RealtimeModel` uses `gemini-3.1-flash-live-preview`**.

Plugin log signatures (from Railway, call `call-_+6587528516_XwbNh8W3js2h` at 20:06:35.81):

```
WARN  'gemini-3.1-flash-live-preview' has limited mid-session update support.
      instructions, chat context, and tool updates will not be applied until the next session.
WARN  generate_reply is not compatible with 'gemini-3.1-flash-live-preview' and will be ignored.
ERROR failed to generate a reply: generate_reply is not compatible with 'gemini-3.1-flash-live-preview'
```

## Impact

Two call sites in `livekit-agent/src/agent.py` silently no-op:

| File:Line | What it does | User-visible symptom |
|---|---|---|
| `src/agent.py:755` | `session.generate_reply(instructions="Greet the caller now.")` — fired right after `session.start()` | Agent is silent on SIP connect; only greets after caller says something first. Confirmed in UAT transcript. |
| `src/agent.py:712` | `session.generate_reply(instructions="Additional intake questions …")` — fired after DB query returns tenant-configured intake_questions | Tenant-configured intake questions never reach the model; callers are never asked them. |

Both regressions were introduced by Phase 63 (1.5.6 enforces a capability-based routing matrix that disallows `generate_reply` for this model — same PR #5413 that motivated the upgrade). Phase 63's pre-merge code audit verified API surface existence but did NOT verify per-model compatibility matrices — flagged in memory `reference_livekit_generate_reply_gemini31.md`.

## What is NOT in scope for this phase

## Candidate approaches (for discuss-phase to pressure-test)

1. **Pure system-prompt greeting** — remove `src/agent.py:755` entirely; add a "START BY GREETING THE CALLER" directive to the system prompt (`build_system_prompt` in `src/prompt.py`) so `RealtimeModel(instructions=system_prompt)` handles it at session start. Risk: unclear whether Gemini 3.1 Flash Live will fire proactively on SIP connect or wait for server-VAD trigger.

2. **Synthetic user turn** — remove `session.generate_reply`; push a fake user turn via `session.input.transcription` (or equivalent 1.5.6 API) to trigger the model. Risk: may not be an API on 1.5.6; requires validation.

3. **Construction-time intake injection** — for `src/agent.py:712`, move intake questions into `system_prompt` at `build_system_prompt` call time (intake questions are known at DB-query completion, which happens BEFORE `session.start()` if we reorder). Requires DB query to complete before session starts — trade-off: 300-500ms added connect latency vs. feature correctness.

4. **Hybrid** — #1 for greeting, #3 for intake. Likely the pragmatic answer.
