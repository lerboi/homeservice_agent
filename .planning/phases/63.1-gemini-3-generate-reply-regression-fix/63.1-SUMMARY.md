## Lessons

1. **Gemini 3.1 capability-guards fail silently, not loudly.** The 1.5.6 google plugin no-ops `generate_reply` on incompatible models with only a warning log line — call sites don't raise, so imports, unit tests, and session startup all succeed while live calls regress. Offline test coverage cannot catch this class of bug without a regression guard that statically scans for the disallowed call pattern (→ `tests/test_no_generate_reply_in_src.py`). Any future 1.5.6 + Gemini-3.x API surface that the plugin hides behind a capability check needs a similar grep-guard.

2. **`session.generate_reply(instructions=...)` on live Gemini 3.1 is unavailable — and synthetic-user-turn injection is unsafe at 1.5.6.** The only reliable way to elicit an agent-first turn at this SDK level is to let Gemini server VAD fire on the caller's first audio frame with an outcome-shaped directive already in the initial `instructions=system_prompt` payload. Verbose verbatim scripts (`"Thank you for calling..."`) are counterproductive — the realtime model is better served by describing what the caller hears ("the FIRST thing the caller hears is a warm, business-branded greeting") and letting it synthesize the prose using interpolated `business_name` + `recording_disclosure`.

3. **Pre-session data injection is the generalizable pattern.** Any tenant- or caller-specific data that used to arrive via `session.generate_reply(instructions=...)` mid-session must now be fetched pre-`session.start()` and threaded through a new `build_system_prompt(kwarg=...)` kwarg. The intake_questions hoist in this phase mirrors the existing pre-session customer_context fetch (Phase 55/56). Future needs (onboarding nudges, per-call coaching, multi-tenant-specific overrides) should follow the same pattern.

4. **`_run_db_queries` index-based result extraction is a latent bug source.** When removing `intake_task` from the parallel `asyncio.gather(...)`, the index-based `results[N]` access pattern had to be rewritten to named-tuple unpacking (`sub_res, call_res, appts_res, events_res, zones_res, buffers_res, blocks_res`) to avoid off-by-one reindexing errors. Named unpacking is self-documenting and refactor-safe; treat index access as a code smell in any future `asyncio.gather` result consumer.

5. **Parallel-executor worktrees on Windows can create stale bases.** Both Plan 01 and Plan 02 executed in parallel against the same `phase-63.1-generate-reply-fix` branch; Plan 01's grep-guard test landed GREEN on first run because Plan 02's deletion commits had already landed. The grep-guard was validated functionally via a synthetic offender. The orchestrator also had to cherry-pick Plan 01's SUMMARY.md onto main because the worktree base had drifted — known Windows EnterWorktree issue. Mitigation: explicit `git reset --soft <expected-base>` check at executor start; cherry-pick recovery path on orchestrator merge.
