**Researched:** 2026-04-24
**Domain:** livekit-agents 1.5.6 + livekit-plugins-google 1.5.6 (gemini-3.1-flash-live-preview) — session-lifecycle + prompt plumbing
**Confidence:** HIGH (every claim verified against installed package source at `C:/Users/leheh/AppData/Roaming/Python/Python313/site-packages/livekit/**` — version 1.5.6, bit-identical to GitHub tag `livekit-agents@1.5.6`; sibling repo at `C:/Users/leheh/.Projects/livekit-agent/` read for code targets)

## Summary

The two `session.generate_reply(...)` call-sites at `livekit-agent/src/agent.py:712` (intake injection) and `:755` (opening greeting) are **both silently broken** on 1.5.6 for `gemini-3.1-flash-live-preview`. The plugin guards `generate_reply` with `if not self._realtime_model.capabilities.mutable_chat_context: ... logger.warning(...); return fut.set_exception(...)` at `realtime_api.py:707-715` — and the capability flag is set to `False` specifically for "3.1" models at `realtime_api.py:289` (`mutable = "3.1" not in model`). The futures resolve with `llm.RealtimeError` that no code path in `agent.py` awaits — so today the regression is a silent no-op plus one log line per call.

**Critical finding from deep plugin audit:** There is **no public API on 1.5.6 that can push a synthetic user turn mid-session** with Gemini 3.1 and reliably trigger an agent reply. All four candidate paths fail:

1. `session.generate_reply(instructions=...)` — hard-rejected at `realtime_api.py:707` (capability guard).
2. `session.generate_reply(user_input="...")` — routes through `agent_activity.py:2826-2853` which calls `_rt_session.update_chat_ctx(chat_ctx)` (hard-skipped for "3.1" at `realtime_api.py:628` via the same `mutable_chat_context=False` gate) **and then** `_rt_session.generate_reply(instructions=...)` which fails as in (1).
3. `session.commit_user_turn()` — calls `self._session.generate_reply()` internally at `agent_activity.py:1285` (blocked), **and** relies on `commit_audio()` which is an explicit noop (`realtime_api.py:1379-1380`: `logger.warning("commit_audio is not supported by Gemini Realtime API.")`).
4. `Agent(chat_ctx=<seed with synthetic user message>)` — initial-history seeding at session connect does fire `send_client_content`, **but with `turn_complete=False`** (`realtime_api.py:855-858`) — this is documented seed-only behavior that does NOT trigger a Gemini response.

**Implication for D-01 hybrid approach:** The "synthetic user turn" leg (Approach #2) is **structurally unavailable** on 1.5.6 + Gemini 3.1. The fallback codified in CONTEXT D-01 ("if the synthetic-user-turn API does not exist on 1.5.6, fall back to Approach #1 for greeting only") is not a fallback — it is **the only path**. Plan 63.1 ships as two independent fixes: (A) intake-questions construction-time injection via `build_system_prompt`, and (B) prompt-directive-only opening greeting. Both `session.generate_reply(...)` calls are deleted outright.

**Greeting behavior under prompt-only approach:** Gemini Live's server VAD fires on the first audio frame from the SIP participant (ambient room tone, breath, "hello?" — anything above the LOW-sensitivity threshold we configure in `realtime_input_config`). With an outcome-shaped directive at the top of the system prompt ("When the caller connects, the first thing the model says is a warm business-branded greeting"), Gemini will greet on the first turn. The **residual risk** is a caller who is fully silent at pickup — they will hear silence until they make any sound. In SIP practice this is ~0% of calls (PSTN carriers inject ambient noise, and callers almost always say "hello"). The regression UAT (call `call-_+6587528516_XwbNh8W3js2h`) showed the caller saying "Hello" first — which would have been enough to fire server VAD and elicit the greeting, except the broken `generate_reply` poisoned the first turn. Removing it fixes the common case.

**Primary recommendation:** Ship three things: (1) delete both `session.generate_reply(...)` calls in `src/agent.py`; (2) add `intake_questions` tenant-data threading that completes before `session.start()` and flows through `build_system_prompt(... intake_questions=...)`; (3) a greeting directive (locale-parity EN + ES) inserted in `prompt.py` as a new `_build_greeting_directive_section` OR woven into the existing `_build_greeting_section`. Add one grep-based pytest to prevent future `session.generate_reply(` regressions in `src/`.

### The one remaining "clever" angle — and why it's still wrong

**Could we flip to `manual_activity_detection=True` and call `start_user_activity()` to spoof a user turn?**

No. Manual activity detection means WE are responsible for calling `start_user_activity()` / `end_user_activity()` on every real user turn — which disables Gemini's server VAD entirely. That would require a full re-architecting of audio handling and would break barge-in. It also doesn't actually inject any content; it just tells Gemini "a user turn just started" — absent real audio, Gemini has nothing to respond to. Out of scope and counter-productive.

### Why Gemini still greets under Approach #1

When Approach #1 (prompt-directive-only greeting) is shipped:

1. `session.start()` completes; Gemini receives `instructions=system_prompt` which now contains the greeting directive.
2. SIP participant's audio stream is routed to Gemini Live as `LiveClientRealtimeInput(audio=...)` continuously (`realtime_api.py:674-683` — `push_audio` runs on every frame).
3. Gemini's server VAD (configured via our `RealtimeInputConfig` — LOW start/end sensitivity, 400ms prefix, 1500ms silence) fires when the first audible speech or breath crosses the threshold.
4. On that first-turn trigger, Gemini's response is governed by the system prompt's greeting directive — it speaks the greeting.

**Residual failure mode:** a caller who stays fully silent and makes NO sound (not even breath) after pickup. Empirically ~0% of SIP calls — PSTN carriers inject 50-200ms of ambient DTMF/noise at connect, and callers reflexively make throat-clear/breath sounds. The UAT that surfaced this regression (`call-_+6587528516_XwbNh8W3js2h`) actually showed the caller saying "Hello" first — which would have fired VAD and elicited a greeting via Approach #1 had `generate_reply` not poisoned the turn.

- **Delete `session.generate_reply(...)` at `src/agent.py:712-717`** (the intake-injection block, lines 702-717 in total). Replace with the construction-time flow described in §Primary Research Target 2.

### Timing invariants analysis

| Data | Needed by | When available today | When needed under fix | Safe to move? |
|------|-----------|----------------------|------------------------|---------------|
| subscriptions | `remove_participant` if blocked | After `_run_db_queries` completes (background) | Same — can stay background | N/A — no change needed |

### Cleanest refactor

Two viable shapes — planner picks:

**Shape A (minimal — hoist only intake_questions as a synchronous pre-step):**
```python
# BEFORE build_system_prompt (around line 248-268 where customer_context is also fetched):
intake_questions_text = ""
if tenant_id:
    try:
        _intake_res = await asyncio.to_thread(
            lambda: supabase.table("services")
                .select("intake_questions")
                .eq("tenant_id", tenant_id)
                .eq("is_active", True)
                .execute()
        )
        all_q = []
        for s in _intake_res.data or []:
            for q in (s.get("intake_questions") or []):
                if q not in all_q:
                    all_q.append(q)

**Latency impact:** One sequential Supabase round-trip (~100-200ms p50) added to the pre-`session.start()` path. Matches D-02's "unbounded, correctness wins" budget. Acceptable.

**Shape B (parallelize with customer_context fetch):**
Wrap the current customer-context fetch at L248-267 in an `asyncio.gather` that also fires the intake query. Saves ~100ms vs Shape A. Slightly more code churn.

**Recommendation:** **Shape A** for minimal-diff hygiene. The 100ms delta is imperceptible (D-02 budget). Shape B is a micro-optimization that isn't worth the cognitive load during a regression fix.

# at column 0..N (simple heuristic: strip leading whitespace, skip if first
# non-whitespace char is '#').
_PATTERN = re.compile(r"\bgenerate_reply\s*\(")


def _iter_python_files(root: Path):
    for p in root.rglob("*.py"):
        # Skip pycache; nothing else to skip in src/
        if "__pycache__" in p.parts:
            continue
        yield p


def test_no_active_generate_reply_calls_in_src():
    offenders: list[str] = []
    for path in _iter_python_files(SRC_DIR):
        for lineno, raw in enumerate(path.read_text(encoding="utf-8").splitlines(), start=1):
            stripped = raw.lstrip()
            if stripped.startswith("#"):
                continue
            if _PATTERN.search(raw):
                offenders.append(f"{path.relative_to(SRC_DIR.parent)}:{lineno}: {raw.rstrip()}")

    assert not offenders, (
        "Phase 63.1: active `generate_reply(` calls found in src/. "
        "livekit-plugins-google 1.5.6 silently drops generate_reply for "
        "gemini-3.1-flash-live-preview. Remove the call or move it under "
        "a test fixture. Offenders:\n  " + "\n  ".join(offenders)
    )
```

**Why this shape:**
- One test, single assertion, clear failure message embeds the "why".
- Ignores fully-commented-out lines (rudimentary `#` prefix check — adequate; false-positives possible if someone puts `generate_reply(` inside a triple-quoted string, but that's acceptably rare and the test failure message is self-explanatory).
- Scans `src/` only — tests/ may legitimately mock `generate_reply` signatures.
- Zero runtime dependencies on livekit-agents or Gemini SDK — runs in <50ms.

**Pre-commit verification:** After Plan 63.1 deletes both call sites, `pytest tests/test_no_generate_reply_in_src.py` passes empty. If a future engineer types `session.generate_reply(...)` back in, the next `pytest tests/` fails with the exact offender line.

## Architecture Patterns

### Pattern 1: Pre-session data fetch for system-prompt assembly

**What:** Fetch tenant-configured data (intake questions, customer context, working hours) BEFORE calling `build_system_prompt`, so the assembled prompt is fully materialized before `AgentSession(llm=model)` is constructed.

**When to use:** Any per-tenant configuration that must be visible to Gemini at session start and does NOT need to update mid-session. Gemini 3.1 forbids mid-session instruction/context mutation (`mutable_instructions=False`), so this is the ONLY viable pattern.

**Example (existing — precedent set by Phase 55/56):**

```python
# src/agent.py:248-267 — customer_context fetched BEFORE build_system_prompt
customer_context = None
if tenant_id:
    customer_context = await fetch_merged_customer_context_bounded(
        tenant_id, from_number, timeout_seconds=2.5
    )

system_prompt = build_system_prompt(
    locale,
    business_name=business_name,
    ...,
    customer_context=customer_context,
)
```

Intake-questions injection follows this exact pattern — just one more awaited fetch before `build_system_prompt`.

### Pattern 2: Builder-per-section with locale branches in prompt.py

**What:** Each prompt section is a `_build_*_section(locale: str, ...)` function. `build_system_prompt` assembles them in a fixed ordered list.

**When to use:** Every prompt edit. Phase 60.3 Plan 12 closed 15 D7 locale-parity gaps using this pattern; any new directive MUST be locale-parallel.

**Evidence:** `prompt.py:1232-1258` — the `sections = [...]` list.

### Anti-patterns to avoid

- **Mid-session `update_instructions`/`update_chat_ctx`/`generate_reply` on Gemini 3.1.** All three are capability-guarded and silently fail. Prefer construction-time injection.
- **Adding a greeting directive in only one locale.** Phase 60.3 established locale parity as an invariant; any EN-only addition is a regression.
- **Adding a new top-level prompt section when an existing section is the natural home.** `_build_greeting_section` is the natural home for the greeting directive.

## Common Pitfalls

### Pitfall 1: Silent failure mode is easy to miss

**What goes wrong:** `generate_reply` returns a future that resolves with an exception nobody awaits. No stack trace, no test failure, just one log line (`generate_reply is not compatible with 'gemini-3.1-flash-live-preview' and will be ignored.`) per call.

**Why it happens:** `plugins/google/realtime/realtime_api.py:712-714` does `fut.set_exception(...)`; `agent.py` at the two call sites does `session.generate_reply(...)` in bare statement form without `await` or storing the future.

**How to avoid:** Part of the fix IS deletion. The regression test (§Target 4) catches reintroduction.

**Warning signs:** Railway logs show `generate_reply is not compatible` WARN lines. Agent transcript first-turn role is `user` (caller), not `agent`.

### Pitfall 2: Reindexing `_run_db_queries` results

**What goes wrong:** After removing `intake_task` from the parallel gather at L648-653, the index offsets for subscription (0), call (1 was 2), and slot_* (2-6 were 3-7) shift. Missing ANY of the downstream consumer site updates produces a latent bug where `results[N]` reads wrong table.

**Why it happens:** Numeric indexing into the `results` tuple is fragile. `results[0]` reads subscriptions (consumed at L677), `results[2]` reads calls insert (consumed at L695), `results[1]` used to be intake (consumed at L703-717 — the code being deleted).

**How to avoid:** Refactor to named unpacking. Plan task should split:
```python
sub_res, call_res, appts_res, events_res, zones_res, buffers_res, blocks_res = await asyncio.gather(
    sub_task, call_task,
    slot_appts_task, slot_events_task, slot_zones_task, slot_buffers_task, slot_blocks_task,
    return_exceptions=True,
)
```
Then update L677 to `if not isinstance(sub_res, Exception): sub_data = sub_res.data`, L695 to `call_res`, and L661-666 `_safe_data(appts_res)` etc.

**Warning signs:** Railway logs show `[agent] _slot_cache prefetched appts=0 events=0 zones=0 buffers=0 blocks=0` unexpectedly, or subscription blocking logic silently passes blocked tenants. Both are caught by existing tests or UAT.

### Pitfall 3: Greeting directive too prescriptive

**What goes wrong:** A directive like "Say: 'Thank you for calling {business_name}. How may I help you?'" causes Gemini to speak that verbatim regardless of context — including awkward mid-call re-greetings if the directive fires twice.

**Why it happens:** Gemini 3.1 Flash Live matches literal strings in system prompts more aggressively than 2.x (per Phase 60.4 commit `68828d7` learnings — `thinking_level` interaction).

**How to avoid:** Outcome-shaped language. "The first thing the caller hears is a warm branded greeting" — no script, no verbatim string. Let the model compose.

**Warning signs:** UAT transcript shows a robotic verbatim greeting; or agent repeats the greeting mid-call. Discard the verbatim clause and re-prompt.

### Pitfall 4: Locale-parity regression

**What goes wrong:** Greeting directive added only in EN. Spanish callers hear silence at pickup, or hear an EN greeting (because the prompt has no ES fallback, so Gemini defaults to whatever the `language=es-US` STT pin suggests).

**Why it happens:** Builder-per-section rule not followed; single `if locale == "en": ...` block added without the mirror.

**How to avoid:** RED→GREEN tests in `test_prompt_greeting_directive.py` include BOTH `test_greeting_directive_en_contains_outcome_text` AND `test_greeting_directive_es_contains_outcome_text` AND `test_greeting_directive_interpolates_business_name` for EACH locale. Phase 60.3 established this testing pattern.

**Warning signs:** Spanish UAT (or ES speaker calling SG tenant) reports no opening greeting while EN works.
