## Deviations from Plan

### Rule 1 — Bug: Greeting-directive assertions scoped to section, not full prompt

**Found during:** Task 1 initial pytest run.
**Issue:** Initial draft used `build_system_prompt(...)` output for all assertions. Two tests accidentally passed RED because `AcmeCorp` (lowercased) also appears in the TRANSFER section and unrelated builder output, and two `no_verbatim_script` tests failed RED for the wrong reason: Phase 60.3 Plan 03 `_build_call_duration_section` contains `"Thank you for calling Voco — have a great"` as a WRONG-example failure-mode string which is NOT a script-inducing directive but a teaching counter-example. A pure-substring "absent" assertion against the full prompt fails permanently (the guard stays RED after Plan 03 ships) which inverts the test's intent.
**Fix:** Rewrote tests to call `_build_greeting_section(locale, business_name, onboarding_complete, t)` directly with in-repo `src/messages/{en,es}.json` loaded for the `t()` stub — section-scoped assertions isolate the true directive surface and let Plan 03's section extension cleanly flip all 6 tests GREEN.
**Files modified:** `tests/test_prompt_greeting_directive.py` (within Task 1, before commit).
**Commit:** Rolled into `943c9d9`.

### Rule 1 — Bug: Conjoined no-verbatim-script guards with directive-presence marker

**Found during:** Task 1 re-run after scope-fix.
**Issue:** With assertions scoped to `_build_greeting_section`, the two `no_verbatim_script_*` tests passed today (greeting section doesn't contain `"thank you for calling"` or `"gracias por llamar"`) — they're structurally invariant guards that should stay green. But the plan spec requires all 6 tests to be RED today (acceptance criterion: `"6 failed"` in pytest stdout).
**Fix:** Conjoined each guard with a directive-presence assertion (`"branded greeting"` / `"con marca"`). Today: directive marker absent → test RED. After Plan 03: directive marker present AND verbatim script still absent → test GREEN. Preserves both the RED-today requirement and the regression-guard semantics.
**Files modified:** `tests/test_prompt_greeting_directive.py` (within Task 1, before commit).
**Commit:** Rolled into `943c9d9`.

### Parallel-execution race: Plan 02 landed before Plan 01 Task 2

**Found during:** Task 2 pytest run.
**Issue:** The shared `phase-63.1-generate-reply-fix` branch had Plan 02's commits (`3e43bb3` = intake-side deletion, `823aab3` = greeting-side deletion) already present when Task 2's grep-guard test was written. The test specification expected RED output flagging `src/agent.py:712` + `src/agent.py:755` as active offenders — but both sites were already deleted by the parallel Plan 02 executor. Result: test passes GREEN on first run.
**Root cause:** No per-agent worktree separation for livekit-agent changes; all Wave 0 + Wave 1 agents shared the single branch HEAD. Plan 02 (Wave 1, `depends_on: [01]`) raced its work ahead of this plan's Task 2 commit.
**Fix:** Validated grep-guard functional correctness by writing a synthetic offender (`src/_tmp_offender_probe.py` with a `session.generate_reply(...)` call) and re-running the test — it fired RED with exit code 1 and the correct assertion message. Guard is semantically valid; it just landed in GREEN state because the regressions it prevents have already been eliminated on this branch.
