## Color

All values reference CSS custom properties defined in `globals.css`. Light and dark values listed for each.

| Role | Light | Dark | Usage |
|------|-------|------|-------|
| Dominant (60%) background | `--background` = `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` (near-black) | Page background, `bg-background` |
| Secondary (30%) surfaces | `--card` = `oklch(1 0 0)` | `oklch(0.205 0 0)` | Card surfaces, `bg-card` via `card.base` token |
| Muted surface | `--muted` = `oklch(0.97 0 0)` | `oklch(0.269 0 0)` | Icon container backgrounds `bg-muted`, disabled areas |
| Accent (10%) | `--brand-accent` = `#C2410C` | `#FB923C` | Switch thumb (active state), focus rings, "Turn on Invoicing" CTA text link style, inline accent icon background `bg-[var(--brand-accent)]/[0.08]` |
| Accent text on dark bg | `--brand-accent-fg` = `#FFFFFF` | `#0F172A` | Text on accent-filled buttons |
| Destructive | `--destructive` = `oklch(0.577 0.245 27.325)` | `oklch(0.704 0.191 22.216)` | NOT used for the flip-off dialog (it is NOT destructive). Reserved only for true destructive actions (delete, permanently remove). |
| Border | `--border` = `oklch(0.922 0 0)` | `oklch(1 0 0 / 10%)` | Card borders, separator lines |
| Muted foreground | `--muted-foreground` = `oklch(0.556 0 0)` | `oklch(0.708 0 0)` | Feature row descriptions, helper text, chevrons |
| Sidebar (constant) | `--sidebar-bg` = `#0F172A` | `#0F172A` (unchanged) | Sidebar never changes — spec note only |

**Accent reserved for:**
- The active (checked) state of the shadcn `<Switch>` component track (`data-[state=checked]:bg-[var(--brand-accent)]`)
- Focus rings on the Switch element (`focus-visible:ring-[var(--brand-accent)]`)
- The icon container background on QUICK_ACCESS rows (`bg-[var(--brand-accent)]/[0.08]`) — NOT used for the MORE_ITEMS features entry icon (uses `bg-muted` to match sibling settings items)
- The `AlertDialogAction` button in the logout dialog pattern only — NOT used for the flip-off dialog action (see Dialog spec below)

**Flip-off dialog button color:** The confirm button uses the shadcn `AlertDialogAction` default style (`bg-primary text-primary-foreground`) — NOT `bg-destructive`. The action is reversible, so no destructive red. This matches the logout dialog pattern in `DashboardSidebar.jsx` line 181 which uses `bg-[var(--brand-accent)]`.

## Surface-by-Surface Interaction Contract

### Surface 1: `/dashboard/more/features` — New Page

**Layout:** Matches `invoice-settings/page.js` exactly.

```
<div className="space-y-6 pb-10">
  <div>
    <h1 className={`text-xl font-semibold ${heading}`}>Features</h1>
    <Separator className="mt-3" />
  </div>
  <div className={`${card.base} p-6 space-y-0 divide-y divide-border`}>
    {/* Feature rows rendered here — divide-y between rows */}
  </div>
</div>
```

**Feature row anatomy (each feature flag):**
```
<div className="flex items-center justify-between gap-4 py-5 first:pt-0 last:pb-0 min-h-[64px]">
  <div className="flex items-center gap-3 flex-1 min-w-0">
    <div className="flex items-center justify-center h-10 w-10 rounded-lg bg-muted shrink-0">
      <Icon className="h-5 w-5 text-muted-foreground" />
    </div>
    <div className="flex-1 min-w-0">
      <p className={`text-sm font-medium ${heading}`}>{feature.label}</p>
      <p className={`text-xs ${body} mt-0.5`}>{feature.description}</p>
    </div>
  </div>
  <Switch
    checked={isEnabled}
    onCheckedChange={handleToggle}
    disabled={isPending}
    aria-label={`${feature.label} — ${isEnabled ? 'on' : 'off'}`}
  />
</div>
```

**Why `bg-muted` for the icon container (not `bg-[var(--brand-accent)]/[0.08]`):** The more/page.js source shows two distinct patterns. `QUICK_ACCESS` items use the accent background (they are navigation shortcuts to core features). `MORE_ITEMS` settings entries use `bg-muted` (they are configuration panels). The Features page is a settings configuration panel — use `bg-muted`.

**Icon for Invoicing row:** `Zap` from lucide-react — evokes capability/enablement, not documents. Alternative: `ToggleRight` or `Settings2`. NOT `FileText` (that is already the Invoices nav icon and Invoice Settings icon — using it again creates visual confusion).

**No "Beta" or "Experimental" badges.** The feature is complete. No badge markup.

**Page is always accessible** — never gated. Proxy matcher MUST NOT include `/dashboard/more/features` in the invoicing gate list.

**Loading state:** While `isPending` (PATCH in flight), the Switch renders with `disabled` prop. The switch opacity reduces to 60% via shadcn default disabled styling. No separate spinner or shimmer — the Switch disabled state is sufficient signal for a sub-500ms toggle operation.

**Error state on PATCH failure:** Rollback the optimistic update. Show sonner toast: `toast.error('Failed to update. Try again.')` — no inline error state on the row itself.

**Success state on PATCH success:** No toast for turning ON (silent). Toast for turning OFF only if the dialog was confirmed: `toast.success('Invoicing disabled. Re-enable here anytime.')`.

**Dark mode:** All classes use semantic tokens (`bg-card`, `bg-muted`, `text-foreground`, `text-muted-foreground`, `divide-border`). No hardcoded color values. The card border `border-border` adapts automatically. The Switch component from shadcn uses Radix UI which inherits the CSS variable system — full dark mode without extra variants needed.

### Surface 2: `/dashboard/more` index — Conditional hiding

**MORE_ITEMS mutation (when `invoicing = false`):**
- Remove index 6: `{ href: '/dashboard/more/invoice-settings', label: 'Invoice Settings', ... }`
- Remove index 7: `{ href: '/dashboard/more/integrations', label: 'Integrations', ... }` — **only** when invoicing is off (integrations page currently serves accounting software only; calendar connections moved to `/dashboard/more/calendar-connections`). When invoicing is on, show both entries as today.
- Add permanently (never hidden): `{ href: '/dashboard/more/features', label: 'Features', icon: Zap, description: 'Turn optional capabilities on or off' }` — placement: between `Integrations` and `AI & Voice Settings`. When invoicing is off and Integrations is hidden, Features sits immediately after Billing.

**"Features" entry icon container:** Uses `bg-muted` style (settings item, not QUICK_ACCESS). Icon: `Zap` from lucide-react, `text-muted-foreground`.

**QUICK_ACCESS mutation (when `invoicing = false`, mobile only):**
- Both Invoices and Estimates entries are removed from the array.
- When the filtered `QUICK_ACCESS` array is empty, do NOT render the quick-access card block at all (no empty container, no label, no divider).
- Behavior: the "Ask Voco AI" card stays visible on mobile regardless of invoicing flag.

**Visual behavior when list shrinks:** Entries use `divide-y divide-border` within the `card.base` container. When an entry is removed from the DOM, the dividers collapse naturally — no gap or visual artifact. No animation needed on the list mutation (this change is server-rendered; users see the post-flag state on next page load).

**Dark mode:** No additional work. `divide-border`, `bg-muted`, `text-muted-foreground` all have dark variants via CSS variables.

### Surface 3: `DashboardSidebar` — Conditional Invoices entry

**When `invoicing = false`:** The Invoices nav item (`{ href: '/dashboard/invoices', label: 'Invoices', icon: FileText }`) is filtered out of `NAV_ITEMS` before render. Removed from DOM entirely. No placeholder. No disabled state. No tooltip.

**Spacing collapse:** NAV_ITEMS renders into `div.flex-1.space-y-1`. When Invoices entry is gone, the `space-y-1` gap between Calls and More collapses naturally. No visual gap artifact. Do not add `mt-*` compensation.

**Before/After visual sequence:**
- Before (invoicing on): Home, Leads, Calendar, Calls, Invoices, More
- After (invoicing off): Home, Leads, Calendar, Calls, More

**Sidebar does not update reactively while the user is on the page.** Flag state is fetched once per layout render (server-side). The sidebar reflects flag state on next navigation/refresh. This is acceptable — flag toggling is a deliberate settings action, not a real-time event.

**Dark mode:** Sidebar is always `bg-[var(--sidebar-bg)] = #0F172A` in both light and dark modes. No dark: variant needed here.

### Surface 4: `LeadFlyout` — Conditional invoice CTAs

**When `invoicing = false`:** The div wrapping both "Create Invoice" (line 737) and "Create Estimate" (line 746) buttons is removed from the DOM entirely. Any per-lead invoice badge, linked invoice count indicator, or invoice/estimate summary rows are also removed.

**Action area reflow:** When the Create Invoice/Estimate block is absent, the flyout footer/action area collapses vertically to whatever remains. Do not add a placeholder or expand other buttons to fill the space. The reduced CTA area is the correct visual state — it communicates that invoicing is not available without any negative framing.

**No "invoicing disabled" message inside the flyout.** The Features panel is the canonical place to learn about the flag. The flyout simply shows fewer CTAs.

**Dark mode:** Flyout uses `bg-card`, `text-foreground` etc — all theme-aware. No additional dark mode work for the removal path.

### Surface 5: Flip-Off Confirmation `<AlertDialog>`

**Trigger condition:** Only shown when toggling `invoicing` from `true` → `false` AND `invoices.count + estimates.count > 0`. Silent (no dialog) in all other cases.

**Dialog visual treatment:** NOT destructive. Use the same styling as the logout confirmation in `DashboardSidebar.jsx` lines 171–186. The `AlertDialogAction` button uses `bg-[var(--brand-accent)] hover:bg-[var(--brand-accent-hover)]` — the orange brand color, not red.

**Exact copy — all strings are locked:**

| Element | Copy |
|---------|------|
| AlertDialogTitle | `Disable invoicing?` |
| AlertDialogDescription (N invoices, M estimates, both > 0) | `You have {N} invoice(s) and {M} estimate(s) on file. Disabling invoicing hides the invoicing tools from your dashboard — your data is preserved and you can re-enable anytime from this page.` |
| AlertDialogDescription (invoices only, estimates = 0) | `You have {N} invoice(s) on file. Disabling invoicing hides the invoicing tools from your dashboard — your data is preserved and you can re-enable anytime from this page.` |
| AlertDialogDescription (estimates only, invoices = 0) | `You have {M} estimate(s) on file. Disabling invoicing hides the invoicing tools from your dashboard — your data is preserved and you can re-enable anytime from this page.` |
| Cancel button label | `Cancel` |
| Confirm button label | `Disable` |
| Confirm button style | `bg-[var(--brand-accent)] hover:bg-[var(--brand-accent-hover)]` (NOT destructive red) |

**Loading state on confirm:** When the user clicks "Disable" and the PATCH is in flight:
- The `AlertDialogAction` button renders with a `Loader2` spinner icon (`h-4 w-4 animate-spin`) prepended to the "Disable" label.
- Button is `disabled` during the request.
- Pattern: `{isPending && <Loader2 className="h-4 w-4 animate-spin mr-2" />}Disable`

**On PATCH success (disable confirmed):**
1. Close dialog (`setConfirmDialogOpen(false)`).
2. Update local state (optimistic update was already applied).
3. Show toast: `toast.success('Invoicing disabled. Re-enable here anytime.')`

**On PATCH error (disable failed):**
1. Close dialog.
2. Rollback optimistic update (switch returns to `checked=true`).
3. Show toast: `toast.error('Failed to disable invoicing. Try again.')`

**On Cancel:** Dialog closes. Switch returns to `checked=true` (no state change). No toast.

**Dialog dark mode:** shadcn `AlertDialogContent` uses `bg-popover text-popover-foreground` via Radix — inherits dark mode automatically. The `AlertDialogDescription` uses `text-muted-foreground` — correct in both modes.

### Surface 6: "Features" entry in More menu — More menu placement

**Position in MORE_ITEMS array:** Between "Billing" and "AI & Voice Settings". Final array order:

```
Services & Pricing
Working Hours
Service Zones & Travel
Notifications & Escalation
Call Routing
Billing
Features            ← new, permanent
[Invoice Settings]  ← hidden when invoicing = false
[Integrations]      ← hidden when invoicing = false
AI & Voice Settings
Account
```

**Rationale for placement:** "Features" is a capability control — it belongs near settings that affect what the product does, not what the account looks like. Billing is a natural anchor before capability controls. Placing Features above Invoice Settings also creates a logical sequence: "turn on features here, then configure them below."

**No section separator or label** distinguishing "Features" from the rest of MORE_ITEMS. All items share one `card.base` with `divide-y divide-border`. Adding a sub-section label would require refactoring the static array pattern; the phase scope does not warrant it.
