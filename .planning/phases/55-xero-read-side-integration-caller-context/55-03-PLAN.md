<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /api/integrations/xero/auth | Owner-initiated; requires authenticated session |
| Xero → /api/integrations/xero/callback | Untrusted redirect; protected by HMAC-signed `state` |
| Browser → /api/integrations/disconnect | Owner-initiated; requires authenticated session + correct tenant |
| Disconnect → Xero revoke endpoint | Outbound HTTPS with refresh_token in body |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-03-01 | Spoofing | OAuth callback CSRF | mitigate | P54's `signOAuthState`/`verifyOAuthState` (HMAC over `SUPABASE_SERVICE_ROLE_KEY`) is reused. Bad state → 400. Test asserts. |
| T-55-03-02 | Tampering | Disconnect for wrong tenant | mitigate | P54 disconnect route uses `getTenantId()` from authenticated session; query is scoped `eq('tenant_id', tenantId).eq('provider', provider)`. Untouched. |
| T-55-03-03 | Information Disclosure | refresh_token in revoke logs | mitigate | Revoke errors logged WITHOUT the token value (only `err.message`). XeroAdapter.revoke (P54 impl) is responsible for not echoing the token in its own error chain. |
| T-55-03-04 | Repudiation | Owner claims they didn't disconnect | accept | Disconnect is a low-stakes destructive op (reversible by re-Connect). No audit log gate required for v6.0. |
| T-55-03-05 | Information Disclosure | Stale cached caller context surviving disconnect | mitigate | `revalidateTag('${provider}-context-${tenantId}')` wipes the broad tag on disconnect, plus `revalidateTag('integration-status-${tenantId}')` wipes the status cache. |
| T-55-03-06 | Information Disclosure | Stale cached context surviving reconnect to a DIFFERENT Xero org | mitigate | Callback also calls `revalidateTag('xero-context-${tenantId}')` so a switch from Xero Org A to Xero Org B doesn't return Org A's data from cache. |
</threat_model>
