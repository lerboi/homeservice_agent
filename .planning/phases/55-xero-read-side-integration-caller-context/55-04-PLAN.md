<objective>
Create `/api/webhooks/xero/route.js` — the inbound endpoint Xero hits when invoice/payment events occur. The handler:

1. **Reads raw body via `request.text()`** (HMAC requires raw bytes; never `request.json()` first).
2. **Verifies HMAC-SHA256** of the raw body against `XERO_WEBHOOK_KEY` env var, comparing with `crypto.timingSafeEqual`. Bad signature → 401. This handles Xero's intent-verify handshake (3 bad probes expect 401, 1 good probe expects 200) automatically.
3. **Parses payload** (only after sig verified). Iterates `payload.events`. For each event:
   a. Resolves the Xero org `event.tenantId` to a Voco tenant via `accounting_credentials.xero_tenant_id`. If no match → silent-ignore (D-07 — prevents Xero retry storms for events we don't care about).
   b. If event is INVOICE category and `resourceId` present, fetch the invoice → fetch the contact → extract `phones[].phoneNumber`. (Extra Xero round-trip OK on webhook path per D-06; never on call hot path.)
   c. For each resolved E.164 phone: `revalidateTag('xero-context-${vocoTenantId}-${phoneE164}')`.
   d. If resolution fails or contact has no phones: `revalidateTag('xero-context-${vocoTenantId}')` (broad fallback per D-06).
4. **Always returns 200** when signature was valid (even on resolution failure / unknown tenant) — Xero retries on non-200.

Default event scope per Open Question 5 + RESEARCH recommendation: INVOICE-only (covers AUTHORISED, PAID, VOIDED state changes which drive `outstandingBalance`). CONTACT events deferred (the 5-min cache TTL self-heals phone changes).

Idempotency per Open Question 5 + RESEARCH recommendation: skip dedicated `xero_webhook_events` table for P55. `revalidateTag` is idempotent; volume is low. Add the table in P58 if telemetry shows duplicate processing.

Purpose: XERO-03. Without this, the cached caller-context goes stale until the 5-min TTL expires — owners would see "I owe what?" surprise scenarios where the AI quotes a balance the customer just paid.
Output: Production-ready webhook route + unit tests + fixtures for intent-verify handshake.

```javascript
import crypto from 'node:crypto';
import { revalidateTag } from 'next/cache';
import { createClient } from '@supabase/supabase-js';
import { getIntegrationAdapter, refreshTokenIfNeeded } from '@/lib/integrations/adapter';

export const runtime = 'nodejs';      // crypto.timingSafeEqual + Buffer require Node runtime
export const dynamic = 'force-dynamic';

/**
 * Xero webhook handler — Phase 55 (XERO-03).
 *
 * Signature verification:
 *   - Read raw body via request.text() (NEVER request.json() first; HMAC needs raw bytes)
 *   - HMAC-SHA256(rawBody, XERO_WEBHOOK_KEY) compared to x-xero-signature header (base64)
 *   - timingSafeEqual to prevent timing attacks
 *   - 401 on bad/missing sig (Xero intent-verify expects this for 3 of 4 probes)
 *   - 200 on good sig (4th probe) — even if event resolution fails downstream
 *
 * Cache invalidation per D-05/D-06:
 *   - Per-phone tag when invoice→contact→phone resolves
 *   - Broad tenant tag fallback otherwise
 *
 * D-07 silent-ignore for unknown Xero tenants prevents Xero retry storms.
 */
export async function POST(request) {
  const rawBody = await request.text();
  const sig = request.headers.get('x-xero-signature');

  if (!sig || !process.env.XERO_WEBHOOK_KEY) {
    return new Response('', { status: 401 });
  }

  const expected = crypto
    .createHmac('sha256', process.env.XERO_WEBHOOK_KEY)
    .update(rawBody, 'utf8')
    .digest('base64');

  let valid = false;
  try {
    const a = Buffer.from(sig, 'utf8');
    const b = Buffer.from(expected, 'utf8');
    valid = a.length === b.length && crypto.timingSafeEqual(a, b);
  } catch {
    valid = false;
  }
  if (!valid) {
    return new Response('', { status: 401 });
  }

  // Sig valid — parse payload. Malformed JSON: still 200 (we accepted the sig, body is Xero's problem).
  let payload;
  try {
    payload = JSON.parse(rawBody);
  } catch {
    return new Response('', { status: 200 });
  }

  const events = Array.isArray(payload?.events) ? payload.events : [];
  if (events.length === 0) {
    // Intent-verify probe 4 has empty events — return 200, no work.
    return new Response('', { status: 200 });
  }

  const admin = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL,
    process.env.SUPABASE_SERVICE_ROLE_KEY,
  );

  for (const event of events) {
    const xeroOrgId = event?.tenantId;        // Xero org UUID (Pitfall 3 — distinct from Voco tenant)
    if (!xeroOrgId) continue;

    // Resolve Xero org → Voco tenant
    const { data: cred } = await admin
      .from('accounting_credentials')
      .select('*')
      .eq('provider', 'xero')
      .eq('xero_tenant_id', xeroOrgId)
      .maybeSingle();
    if (!cred) continue;                       // D-07 silent-ignore unknown tenant

    const vocoTenantId = cred.tenant_id;
    let phones = [];

    if (event?.eventCategory === 'INVOICE' && event?.resourceId) {
      try {
        const adapter = await getIntegrationAdapter('xero');
        const refreshed = await refreshTokenIfNeeded(admin, cred);
        adapter.setCredentials(refreshed);

        // Fetch invoice → contact → phones
        const invResp = await adapter._xeroClient.accountingApi.getInvoices(
          xeroOrgId,
          undefined,
          undefined,
          undefined,
          [event.resourceId],                // ids
        );
        const inv = invResp.body?.invoices?.[0];
        const contactID = inv?.contact?.contactID;
        if (contactID) {
          const contactsResp = await adapter._xeroClient.accountingApi.getContacts(
            xeroOrgId,
            undefined,
            undefined,
            undefined,
            [contactID],                     // ids
          );
          const contact = contactsResp.body?.contacts?.[0];
          phones = (contact?.phones || [])
            .map(p => (p?.phoneNumber || '').trim())
            .filter(Boolean);
        }
      } catch (err) {
        // Resolution failed (Xero 5xx, rate limit, refresh failure, etc.)
        // Fall through to broad tag invalidation.
        phones = [];
      }
    }

    if (phones.length === 0) {
      revalidateTag(`xero-context-${vocoTenantId}`);
    } else {
      for (const p of phones) {
        revalidateTag(`xero-context-${vocoTenantId}-${p}`);
      }
    }
  }

  return new Response('', { status: 200 });
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Xero servers → /api/webhooks/xero | Untrusted POST from public IP space; HMAC is the only auth |
| /api/webhooks/xero → Xero API | Outbound HTTPS for invoice→contact→phone resolution |
| /api/webhooks/xero → Postgres | Service-role read of accounting_credentials |
| /api/webhooks/xero → Next.js cache | revalidateTag dispatches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-04-01 | Spoofing | Forged webhook from non-Xero source | mitigate | HMAC-SHA256 of raw body against `XERO_WEBHOOK_KEY`; bad sig → 401. Test asserts. |
| T-55-04-02 | Information Disclosure | Timing-attack on signature compare | mitigate | `crypto.timingSafeEqual` (NOT `===`). Length check before timingSafeEqual to avoid throw. |
| T-55-04-03 | Tampering | Webhook replay | accept | P55 omits dedicated idempotency table (Open Question 5 → defer to P58). `revalidateTag` is idempotent so replay invalidates the same tag twice — no harm. |
| T-55-04-04 | Denial of Service | Unbounded payload | mitigate | Next.js default body size limit applies (~1MB). Long event arrays processed in O(n) but each resolution is bounded by Xero rate limits which throttle the source. |
| T-55-04-05 | Denial of Service | Unknown-tenant flood (attacker sends valid HMAC for orgs we don't manage) | mitigate | D-07 silent-200 prevents Xero retry storms; the lookup is a single indexed query (cheap). Attacker would need our signing key — which is the moat. |
| T-55-04-06 | Information Disclosure | Cross-tenant cache leak via attacker-controlled phones in cacheTag | mitigate | Phones come from Xero contact records (server-side), NOT from request body or attacker input. |
| T-55-04-07 | Tampering | OData injection via resourceId/contactID | mitigate | resourceId/contactID are passed as `ids` array param (NOT interpolated into where filter); xero-node SDK serializes safely. |
| T-55-04-08 | Information Disclosure | XERO_WEBHOOK_KEY leak via error logs | mitigate | Catch blocks log only `err.message`; never log `sig`, `expected`, `process.env.XERO_WEBHOOK_KEY`, or full credential rows. |
